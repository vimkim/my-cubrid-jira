# CBRD-26912 쉽게 읽기 — "이 컬럼은 가능하면 인라인에 남겨줘" 기능

> 이 문서는 신입사원이 CBRD-26912 를 처음 볼 때를 위한 **쉬운 설명본**입니다.
> 원본(JIRA 업로드용 정식 문서)은 `issues/CBRD-26912-oos-prefer-inline.md` 에 있습니다.
> 이 파일은 `onboarding/` 폴더에 있어서 JIRA 로 업로드되지 않습니다. 마음껏 읽고 고쳐도 안전합니다.

---

## 0. 30초 요약 (TL;DR)

- CUBRID 는 한 행(row)이 너무 커지면, **큰 컬럼 값을 따로 떼어 다른 곳(OOS)에 보관**합니다. 이걸 "demote(강등)" 라고 부릅니다.
- 지금은 떼어낼 컬럼을 **오로지 크기 순서**(큰 것부터)로만 고릅니다. "이 컬럼은 자주 읽으니까 가능하면 떼지 말아줘" 라고 말할 방법이 없습니다.
- 이 이슈는 SQL 에 `STORAGE PREFER_INLINE` 옵션을 추가해서, 사용자가 컬럼에 **"가능하면 인라인에 남겨줘"** 라는 의사를 표시할 수 있게 합니다.
- 핵심은 **soft(부드러운)** 동작입니다. "절대 떼지 마"가 아니라 **"맨 마지막에 떼어줘"** 입니다. 그래서 기존 규칙("행은 언제나 페이지 안에 들어간다")이 깨지지 않습니다.

---

## 1. 먼저 알아야 할 용어들 (그림으로 이해하기)

이 기능을 이해하려면 단어 몇 개만 알면 됩니다. 어려운 단어가 나와도 겁먹지 마세요. 비유로 설명합니다.

### 🧳 행(record) 과 페이지(page)

데이터베이스는 한 행(예: 회원 한 명의 정보)을 **하나의 "레코드"** 로 디스크에 저장합니다.
레코드는 **"페이지"** 라는 고정 크기 상자 안에 들어가야 합니다.

> **비유:** 레코드 = 여행 가방에 담은 짐, 페이지 = 비행기 기내 반입 가방의 최대 크기.
> 짐은 반드시 가방 안에 들어가야 합니다. CUBRID 의 페이지 크기는 보통 16KB 이고,
> "한 행은 페이지의 1/4(=4KB) 까지" 라는 기준선을 둡니다. 이 선을 넘으면 손을 봐야 합니다.

### 📦 인라인(inline) vs OOS

- **인라인(inline)**: 컬럼 값이 레코드 안에 **그대로 같이** 들어 있는 상태. (= 짐을 가방 안에 넣음)
- **OOS (Out-of-row Storage)**: 큰 컬럼 값을 레코드에서 **떼어내 별도의 파일에 따로** 보관하는 것. 레코드에는 "그 값은 저쪽에 있어요" 라는 주소(포인터)만 남습니다. (= 부피 큰 짐은 따로 화물칸/창고로 보냄)

> **왜 OOS 를 쓰나?** 큰 컬럼이 레코드 안에 그대로 있으면, **작은 컬럼 하나만 읽고 싶을 때도** 그 큰 덩어리를 통째로 디스크에서 읽어야 해서 느립니다. 큰 값을 밖으로 빼두면, 작은 컬럼만 읽을 때 빨라집니다.

### ⬇️ demote (강등)

레코드가 너무 커서(4KB 초과) 어떤 컬럼을 인라인 → OOS 로 **떼어내는 동작**을 "demote(강등)" 라고 합니다.

> **비유:** 가방이 안 닫혀서, 부피 큰 물건을 하나씩 꺼내 창고로 보내는 일.

### 💸 OOS 의 대가: `oos_read`

OOS 로 보낸 컬럼은 읽을 때마다 "창고에 다녀오는" 추가 디스크 읽기(`oos_read`)가 필요합니다.
즉 **자주 읽는 컬럼을 OOS 로 보내면 손해**입니다. 읽을 때마다 추가 비용이 드니까요.

## 2. 지금 무엇이 문제인가?

레코드가 4KB(`DB_PAGESIZE/4`)를 넘으면, CUBRID 는 떼어낼 후보 컬럼들을 **크기 큰 순서로 줄 세운 다음, 큰 것부터** 창고(OOS)로 보냅니다.

문제는 **고르는 기준이 "크기" 딱 하나**라는 점입니다.

> **예시:**
> 한 행에 이런 두 컬럼이 있다고 합시다.
> - `profile` : 4KB, **자주 조회함** (hot 컬럼)
> - `log` : 3KB, 거의 안 읽음 (cold 컬럼)
>
> 현재 규칙은 "큰 것부터" 이므로, **더 큰 `profile`(4KB) 이 먼저 OOS 로 끌려갑니다.**
> 그 결과 자주 읽는 `profile` 을 조회할 때마다 매번 `oos_read` 비용이 발생합니다. 😩
> 사용자는 "아니, `profile` 은 자주 읽으니까 `log` 를 대신 보내줘" 라고 말하고 싶은데, **그 방법이 없습니다.**

## 3. 추가하려는 기능: `STORAGE PREFER_INLINE`

사용자가 SQL 에서 **컬럼별로** "이건 가능하면 인라인에 남겨줘" 라고 표시할 수 있게 합니다.

```sql
CREATE TABLE t (
  id       INT,
  payload  VARCHAR(4096) STORAGE PREFER_INLINE,  -- "가능하면 떼지 말아줘" (OOS 후순위)
  logs     BIT VARYING   STORAGE DEFAULT          -- 지금까지 하던 대로 (생략과 동일)
);

-- 나중에 ALTER 로도 바꿀 수 있습니다
ALTER TABLE t MODIFY payload VARCHAR(4096) STORAGE DEFAULT;
ALTER TABLE t MODIFY logs    BIT VARYING   STORAGE PREFER_INLINE;
```

- 옵션 값은 두 가지뿐: **`DEFAULT`**(지금처럼 크기순) 과 **`PREFER_INLINE`**(인라인 우선).
- 아무것도 안 적으면 `DEFAULT` 와 같습니다. → **기존 테이블·쿼리는 100% 그대로 동작**합니다.

---

## 4. 가장 중요한 개념: "soft" 가 무슨 뜻인가?

`PREFER_INLINE` 은 **"절대 OOS 로 보내지 마"가 아닙니다.** "보낼 거면 **맨 나중에** 보내" 입니다.
이런 부드러운 방식을 **soft** 라고 부릅니다.

다르게 말하면:
- demote 를 **할지 말지** 를 바꾸는 게 아니라,
- demote 를 한다면 **어떤 컬럼부터** 할지, 그 **순서만** 바꿉니다.

> **비유 (다시 가방):**
> 가방이 안 닫힐 때 꺼낼 물건의 순서를 정하는데,
> `PREFER_INLINE` 물건은 **"맨 마지막에 꺼낼 후보"** 줄 맨 뒤로 보냅니다.
> 다른 물건을 다 꺼내도 가방이 안 닫히면? **그때는 어쩔 수 없이 이 물건도 꺼냅니다.**

이게 왜 좋은가 하면 — **"레코드는 언제나 페이지 안에 들어간다"** 는 CUBRID 의 기존 약속(불변식)이 그대로 지켜집니다. `PREFER_INLINE` 컬럼도 최후의 수단으로는 demote 후보에 끝까지 남아 있기 때문입니다. 그래서 INSERT 가 "공간 없음" 으로 실패하는 일이 생기지 않습니다.

---

## 5. soft 방식의 한계 (이건 의도된 것)

새 직원이 꼭 알아야 할 두 가지 "함정" 입니다.

1. **떼어낼 다른 컬럼이 없으면 효과가 없습니다.**
   큰 컬럼이 딱 하나뿐인 행에서 그 컬럼에 `PREFER_INLINE` 을 걸어도, 행이 4KB 를 넘으면 결국 그 컬럼이 OOS 로 갑니다. **대신 내보낼 컬럼이 없으니까요.** ("무조건 인라인 고정" 은 hard 강제가 필요한데, 이번 이슈 범위 밖입니다 — 10번 참고.)

2. **공짜가 아닙니다. 비용이 옮겨갈 뿐입니다.**
   hot 컬럼을 지키면, 그 대신 OOS 로 가게 된 **다른 컬럼**을 읽을 때 `oos_read` 비용이 듭니다. **지키려는 컬럼이 다른 컬럼보다 더 자주 읽힐 때만** 전체적으로 이득입니다. 그냥 켜면 무조건 빨라지는 마법 스위치가 아닙니다.

---

## 6. ALTER 의 적용 시점 (사용자가 헷갈리기 쉬운 부분)

`ALTER ... STORAGE PREFER_INLINE` 은 **그 이후의 INSERT/UPDATE 부터** 적용됩니다.

- 이미 OOS 로 나가 있던 **기존 값들은 ALTER 만으로 인라인으로 돌아오지 않습니다.** (되돌리려면 해당 행을 다시 써야 합니다. → 자동 재배치는 후속 이슈)
- `ALTER ... MODIFY` 에서 **`STORAGE` 절을 생략하면, 기존 설정이 그대로 유지**됩니다 (DEFAULT 로 초기화되지 않음). 명시적으로 되돌리고 싶으면 `STORAGE DEFAULT` 라고 적어야 합니다.

> 참고: 이 "생략하면 기존값 유지" 동작은 PostgreSQL 의 `SET STORAGE`, 그리고 CUBRID 의 기존 `INVISIBLE` 컬럼 옵션과 똑같은 방식입니다.

## 7. 내부 동작이 궁금하다면 (선택 읽기)

> 여기는 구현/리뷰 단계에서 보면 됩니다. triage 만 할 때는 건너뛰어도 됩니다.

### 플래그 한 비트가 흘러가는 길

이 기능은 사실상 **"플래그 1비트를 SQL 파서부터 저장 엔진까지 전달"** 하는 일입니다.
이미 `INVISIBLE` 컬럼 옵션이 똑같은 길을 다니고 있어서, 그 선례를 그대로 따라갑니다.

```
[SQL 파싱]   STORAGE PREFER_INLINE 이라고 입력
   → 파서가 PT 노드에 "PREFER_INLINE" 으로 기록
[스키마 저장] 컬럼 메타데이터(SM_ATTRIBUTE)에 비트 한 개를 켬 (SM_ATTFLAG_OOS_PREFER_INLINE = 0x1000)
[직렬화]     기존 코드가 알아서 통째로 저장/전송 (이 부분은 수정 불필요)
[복원]       서버에서 다시 읽을 때 OR_ATTRIBUTE 의 비트필드 oos_prefer_inline(1비트) 로 풀어냄
[실제 사용]  heap_file.c 의 demote 정렬에서 이 비트를 봄  ← (!) 진짜 동작이 바뀌는 유일한 지점
```

`(!)` 한 곳만 실제 정책이 바뀌고, 나머지는 전부 그저 **비트를 실어 나르는 배선** 일 뿐입니다.
카탈로그(시스템 테이블) 디스크 포맷은 **안 바뀝니다.** 그래서 기존 데이터와 완전히 호환됩니다.

### 정렬 규칙이 어떻게 바뀌나 (핵심 한 곳)

- **지금:** 후보를 `{크기}` 기준으로 큰 것부터 정렬 → 큰 것부터 demote.
- **바뀐 후:** 정렬 1순위 키를 하나 추가합니다.
  1. **PREFER_INLINE 인가?** — 아닌 컬럼(일반)을 먼저, PREFER_INLINE 컬럼을 **맨 뒤로**.
  2. 같은 그룹 안에서는 — **크기 큰 것 먼저** (기존과 동일).
  3. 크기도 같으면 — 컬럼 번호(idx) 순으로 (순서를 항상 똑같이 만들기 위한 tiebreak).

> **3번 tiebreak 가 왜 필요할까?** `std::sort` 는 "같은 값" 들의 순서를 보장하지 않습니다(unstable). 크기가 같은 두 컬럼의 demote 순서가 빌드 환경마다 달라지면 회귀 테스트가 들쭉날쭉(flaky)해집니다. idx 키를 넣으면 항상 똑같은 순서가 나와서, **PREFER_INLINE 을 안 쓴 기존 동작과 비트 단위로 100% 동일**해집니다. (= 회귀 없음)

## 8. PostgreSQL 을 안다면 (참고)

이 기능은 PostgreSQL TOAST 의 `ALTER TABLE ... SET STORAGE MAIN`("가능한 한 인라인 유지")에 대응합니다.
다만 PostgreSQL 의 STORAGE 값(PLAIN/EXTERNAL/EXTENDED/MAIN)을 그대로 가져오지는 않습니다.
PostgreSQL 의 STORAGE 에는 **압축** 의미가 섞여 있는데, CUBRID OOS 에는 압축 개념이 없기 때문입니다(압축은 별도 기능). 그래서 OOS 배치 우선순위에만 한정해 **두 값(DEFAULT / PREFER_INLINE)**으로 단순화했습니다.

---

## 9. 이번 이슈에서 **하지 않는** 것 (후속 이슈로 분리)

- ❌ **hard 강제** (`FORCE_INLINE` 같은, 컬럼을 절대 OOS 로 안 보내는 옵션) — 행이 페이지를 넘으면 별도의 overflow page 경로로 넘어가는데, OOS 와 overflow 가 동시에 동작하는 상황을 따로 검증해야 해서 분리합니다.
- ❌ **기존 데이터 소급 재배치** — ALTER 한다고 이미 OOS 에 있는 값이 자동으로 인라인으로 돌아오지 않습니다.
- ❌ **자동 판단** — 접근 빈도 통계를 보고 알아서 demote 하는 기능.

## 10. 한 문장으로 다시

> **"행이 너무 커서 큰 컬럼을 밖으로 떼어내야 할 때, `STORAGE PREFER_INLINE` 이 붙은 컬럼은 떼어내는 순서의 맨 뒤로 보내서 — 가능하면 인라인에 남게 하는, 부드러운(soft) SQL 힌트."**
