## Presentation Outline

### 1. 도입 (개요·배경)

- Claude Code 개요 (Anthropic, CLI 기반 agentic 도구)
- **Pro vs Max 5x vs Max 20x 비교**

| 항목 | Pro | Max 5x | Max 20x |
|------|-----|--------|---------|
| 월 사용 한도 | 기본 | 약 5배 | 약 20배 |
| 컨텍스트 창 | 표준 | 확장 | 확장 (긴 대화 가능) |
| 월 가격 | $22 | $110 | $220 |

- Pro 는 주로 Sonnet 모델 → CUBRID 개발에는 도움이 거의 안 됐음
- **Max 20x vs API 가성비**: Heavy user 기준 약 10~36배 저렴, 외부 사례 94% 절감

### 2. Part 1 — 코드베이스 탐색·분석 보조

- `heap_file.c` (15,000+ 줄) 레코드 삽입 흐름 파악 (clangd LSP 자동 호출)
- "어디를 봐야 할지" 를 빠르게 지목 — `vector type 추가 위치`, `vacuum 수동 실행 방법`, `varbit vs varchar 효율 비교` 등
- 에러 메시지 → 코드 위치 추적, 타인 PR 리뷰 준비 단축

### 3. Part 2 — 실제 개발 작업

- CI shell-test 실패 분석 (로그 + 스크립트 입력 → 근본 원인 파악)
- isolation test (`.ctl`) 초안 작성, 헬퍼 함수 + 단위 테스트 동시 작성
- **5시간 자율 작업 에피소드**: vector type / DDL / DML 재구현 — 발표자 2개월 작업과 유사한 결과물 산출, 에러·예외 처리까지 포함
- **Coredump 디버깅 자동화 루프**: 빌드 → 실행 → `gdb bt` → 원인 가설 → 수정 → 반복 (NULL deref, 포인터 해제, lock 순서 오류 해결 사례)

### 4. Part 3 — 문서·커뮤니케이션

- JIRA 이슈 / PR description 초안 (팀 컨벤션 준수)
- Greptile bot 리뷰 일괄 응답, 조사 노트 → 내부 문서 변환
- PR 리뷰 코멘트 응답 보조

### 5. 한계 — 다섯 가지 주의점

| # | 한계 | 요지 |
|---|------|------|
| 1 | 검증은 여전히 개발자 몫 | AI 출력은 항상 초안 — 빌드·테스트·리뷰 필수 |
| 2 | 무의미한 TC 생성 | 형식 채우는 테스트, 경계값·NULL 누락 |
| 3 | 통과만 목적인 테스트 | mock 과다 사용, 실제 로직 미실행 |
| 4 | 생성된 코드가 느릴 수 있음 | 피드백 루프에 성능 벤치마크 없으면 비효율 자료구조 선택 |
| 5 | AI 피로도 (Brain Fry) | 대기 시간 병렬화 → 컨텍스트 스위칭 비용, 신체적 피로 |

### 6. 작업 흐름의 변화

- **인간의 Feedback Loop 도 가속** — 각 단계마다 더 명확하고 빠른 피드백
- 이전: 가설 → 조사 → 코드 분석 → 설계 → 구현 → 개발자 검증 → TC 생성 → 스펙 변경
- 지금: 가설 → **Claude 로 빠른 POC 검증** → 설계 → 스펙 검증 → unit test · TC 작성 → TC 검증 → AI 가 구현 → 코드 리뷰

### 7. 조심스럽게 예측해보는 개발 프로세스 변화

- **SDD (Spec Driven Development)** — 스펙을 먼저 글로 명확히 작성한 뒤 구현
- **SSOT (Single Source of Truth)** — JIRA 이슈를 최대한 자세히 작성, Claude Code 가 이슈를 직접 읽고 구현 자동 수행하는 흐름으로 진화 예상
- **TDD (Test Driven Development)** — 실패 테스트 → 통과 구현, AI 검증 기준 제공에 효과적

### 8. 시작 가이드 + Skill · Plugin 전략

- 바로 적용 가능한 세 가지: 낯선 코드 탐색 · 디버깅 첫 걸음 · 문서 초안
- **Skill — "가장 개인적인 것이 가장 창의적이다"**
  - 범용 skill 은 언젠가 누군가가 만든다 → 진짜 가치는 **팀·개인 워크플로 특화 skill**
  - `/learner` → `/skill-creator` 흐름으로 오래 걸린 작업을 학습·재사용 가능하게 등록
- **Skill → Script → Plugin 화** — 토큰 절약 + 재현성
  - 배포 계층: **개인 → 팀 → 연구개발본부 플러그인**
  - 장기 제안: **플러그인 관리 업무를 공식 업무로 추가**

### 9. 마무리 메시지 + Q&A

- **사람에게 친화적인 업무 = AI 에게 친화적인 업무** — Win-win
- `ctp.sh --help`, `README`, 빌드 스크립트 주석 개선 → 신규 입사자 온보딩과 AI 효율이 **같은 방향** 으로 개선
- **암묵지 → 명시지 전환** 의 가치 — 에러 발생 시 참고 로그 위치 등을 문서화
- **AI 자동화의 가장 큰 적: Hang · 무한 루프 · 끝을 모르는 작업** — iteration/시간 warning, timeout 짧게, 대안 경로 설계
