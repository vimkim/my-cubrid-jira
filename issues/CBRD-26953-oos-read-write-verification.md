# OOS 읽기/쓰기 동작 검증 보고서

> 작성일: 2026-06-18 | 브랜치: `feat/oos` | 검증 빌드: debug
> 검증 모드: **SA(`csql -S`) + CS(실서버 `cub_server` 접속) 양쪽 — 결과 완전히 동일**
> 한 줄 요약: **필요 없는 컬럼은 절대 읽지 않는다** 는 것을, 직접 실행해서 두 가지 방법으로 확인했습니다.

---

## 0. 이 문서를 읽으면 알게 되는 것

- OOS 가 무엇인지 (3분 설명)
- "OOS 컬럼을 조회하지 않으면 OOS 파일을 읽지 않는다"가 정말인지
- `SELECT` / `UPDATE` 할 때 OOS 파일을 **몇 번** 읽고 쓰는지
- 그걸 우리가 **어떻게 증명** 했는지 (`csql -S` + `gdb` + `oos.log`)
- **SA 모드(`csql -S`) 뿐 아니라 CS 모드(실서버 `cub_server` 접속)에서도 횟수가 같은지** (8절)

---

## 1. OOS 가 뭔가요? (아주 쉬운 비유)

테이블의 한 행(row)은 보통 **한 묶음** 으로 디스크에 저장됩니다.
그런데 어떤 컬럼은 값이 아주 큽니다 (예: 긴 텍스트, 큰 바이너리).

```
일반 저장 방식:
[ id(4B) | 이름(20B) | 큰_텍스트(5000B) | 큰_사진(5000B) ]   <- 한 행 전체가 ~10KB
```

문제: `SELECT id FROM ...` 처럼 **작은 값 하나만** 보고 싶어도,
디스크에서 행을 읽으면 **큰 값까지 통째로** 따라옵니다. 낭비죠.

**OOS (Out-of-row Overflow Storage)** 는 이 큰 컬럼들을 본체에서 빼내
**별도 창고(OOS 파일)** 에 보관하고, 본체에는 "창고 주소표(OOS OID, 16바이트)"만 남깁니다.

```
OOS 방식:
[ id(4B) | 이름(20B) | 주소표(16B) | 주소표(16B) ]   <- 본체는 작아짐 (~50B)
                          |            |
                          v            v
                    [ 큰_텍스트 ]  [ 큰_사진 ]        <- 별도 OOS 파일(창고)
```

> **비유**: 사무실 책상(heap record)에는 서류 색인표만 두고,
> 부피 큰 문서는 창고(OOS 파일)에 보관합니다.
> 색인표만 필요하면 창고에 갈 필요가 없습니다.

### 언제 창고로 보내나요? (현재 규칙, CBRD-26776)

1. 행이 **4KB**(= `DB_PAGESIZE/4`)보다 커야 함. 작으면 전부 책상에 그대로 둠.
2. 4KB를 넘으면, **가장 큰 컬럼부터 하나씩** 창고로 보냄.
3. 행이 4KB 아래로 줄어드는 **순간 멈춤**.

> 중요: 그래서 "창고에 안 간 컬럼(=일반 컬럼)"이 **반드시 작은 건 아닙니다.**
> 그냥 "가장 큰 컬럼을 보냈더니 더 보낼 필요가 없어서 남은 것"일 수 있어요.

## 2. 알아야 할 단어 딱 3개

| 단어 | 뜻 (쉽게) |
|------|-----------|
| **OOS 컬럼** | 창고(OOS 파일)로 보내진 큰 컬럼 |
| **일반 컬럼 (inline)** | 책상(heap record)에 그대로 남은 컬럼 (`id` 같은 작은 값 포함) |
| **`oos_read`** | 창고에서 값 하나를 꺼내오는 함수 (= "창고에 다녀옴") |
| **`oos_insert`** | 창고에 값 하나를 새로 보관하는 함수 |
| **`oos.log`** | 창고 출입 기록부 (debug 빌드에서 `$CUBRID/log/oos.log` 에 남음) |

이 보고서의 핵심 질문은 한마디로:
> **"창고에 안 가도 되는 상황에서, 정말 창고(`oos_read`)에 안 가는가?"**

## 3. 팀에서 받은 질문 (검증 목표)

```sql
CREATE TABLE tbl (id INT, non_oos_vc ..., oos_vc ...);   -- oos_vc 만 창고로 감
SELECT id, non_oos_vc FROM tbl;                          -- 창고 컬럼은 조회 안 함
```

> **질문 1 (SELECT):** 위 쿼리는 `oos_read` 를 **단 한 번도** 부르면 안 된다. 정말인가?
> **질문 2 (UPDATE):** OOS 컬럼 / 일반 컬럼을 수정할 때 `oos_read` 와 `oos_insert` 가 각각 몇 번 일어나나?

---

## 4. 어떻게 증명했나 (3중 확인)

말로 "안 읽습니다"라고 하면 못 믿죠. 그래서 **3가지를 동시에** 썼습니다.

1. **`csql -S`** - 서버를 따로 안 띄우고 한 프로세스에서 SQL 실행 (standalone 모드)
2. **`gdb` 중단점(breakpoint)** - `oos_read`/`oos_insert` 함수에 덫을 놓아, **불릴 때마다 카운트**.
   호출이 0이면 -> 그 함수는 실행되지 않은 것 (가장 확실한 증거)
3. **`oos.log`** - 창고 출입 기록부를 직접 확인

### 가장 중요한 검증 원칙: "양성 대조군"

> "창고에 안 갔다"는 결과가, **덫이 고장 나서** 안 잡힌 건지 헷갈릴 수 있습니다.
> 그래서 먼저 **일부러 창고에 가는 쿼리**(`SELECT oos_vc`)를 돌려서
> **덫이 제대로 작동하는지(= 정말 잡히는지)** 확인했습니다.
> 덫이 잡는 게 증명된 뒤에야, "안 잡혔다 = 진짜 안 갔다"를 믿을 수 있습니다.

(실제로 이 과정에서 함정이 두 개 있었습니다 - 부록 참고. 처음엔 덫이 엉뚱한 프로세스를 보고 있었어요.)

## 5. 테스트용 테이블

```sql
CREATE TABLE tbl (id INT, non_oos_vc BIT VARYING, oos_vc BIT VARYING);
-- 각 행: non_oos_vc = 1KB, oos_vc = 5KB  -> 행 ~6KB > 4KB
```

> `BIT VARYING`(VARBIT)을 쓴 이유: `VARCHAR` 는 내부적으로 압축돼서 크기가 들쭉날쭉합니다.
> VARBIT 은 압축이 없어 **딱 N바이트** 로 저장돼, 창고로 갈지 말지를 정확히 통제할 수 있습니다.

| 컬럼 | 값 크기 | 어디에 저장? | 이유 |
|------|---------|--------------|------|
| `id` | 4B | 책상(inline) | 작음 |
| `non_oos_vc` | 1KB | 책상(inline) | 1KB라 작진 않지만, **더 큰 `oos_vc` 를 창고로 보내니 행이 4KB 아래로 줄어서 멈춤** |
| `oos_vc` | 5KB | **창고(OOS)** | 가장 커서 제일 먼저 창고행 |

**행 개수는 테스트마다 다릅니다** (숫자가 헷갈리지 않도록 미리 정리):
- **SELECT 테스트(6절)**: 행 **2개** -> "행 수만큼 읽는다"를 보여주기 위함
- **UPDATE 테스트(7절)**: 행 **1개** -> 횟수를 깔끔하게 세기 위함

> 각 INSERT 직후 `oos.log` 에 `oos_vc` **보관 기록(행당 1건)** 이 찍히는 것으로
> `oos_vc` 가 진짜 창고에 갔음을 먼저 확인했습니다 (= "양성 대조군"의 출발점).

## 6. 결과 (1) - SELECT

행 2개를 넣고 두 가지 쿼리를 실행했습니다.

| 실행한 쿼리 | `oos_read` 횟수 (gdb) | `oos.log` 기록 | 해석 |
|-------------|:---:|:---:|------|
| `SELECT id, oos_vc`  *(대조군: 일부러 창고 조회)* | **2** | 2 | 행마다 1번씩 = 정상 |
| `SELECT id, non_oos_vc`  *(<- 팀 질문)* | **0** | 0 (파일조차 안 생김) | **창고에 한 번도 안 감!** |

### 결론

> **[결론]** OOS 컬럼을 `SELECT` 목록에 넣지 않으면 `oos_read` 는 절대 호출되지 않습니다. (질문 1 = 사실)

이유는 간단합니다. CUBRID 는 **요청한 컬럼만 골라서** 값을 채웁니다(게으른 읽기, lazy resolve).
`non_oos_vc` 는 책상에 그대로 있으니 바로 읽고, `oos_vc` 는 **목록에 없으니 쳐다보지도 않습니다.**

```
SELECT id, non_oos_vc FROM tbl
     |
     v  요청한 컬럼만 처리
   id         - 책상에서 바로 읽기 (창고 X)
   non_oos_vc - 책상에서 바로 읽기 (창고 X)
   oos_vc     - [X] 요청 안 함 -> 건드리지 않음 -> oos_read 호출 0
```

> 참고 - 부수 효과: 만약 행이 100개면 `SELECT id, oos_vc` 는 `oos_read` 100번,
> `SELECT id, non_oos_vc` 는 **행이 몇 개든 0번** 입니다.

## 7. 결과 (2) - UPDATE (한 행 기준)

이게 조금 의외입니다. 두 가지 UPDATE를 비교했습니다.

| 수정한 컬럼 | `oos_read` | `oos_insert` | 무슨 일이 일어났나 |
|-------------|:---:|:---:|------|
| **`oos_vc`** (창고 컬럼 수정) | **0** | **1** | 새 값을 내가 직접 줬으니 **읽을 필요 없음**. 새 값 1건만 창고에 보관. |
| **`non_oos_vc`** (책상 컬럼 수정, `oos_vc` 는 그대로) | **1** | **1** | `oos_vc` 를 **건드리지도 않았는데** 1번 읽고 1번 다시 보관! |

### 왜 작은 컬럼을 고쳤는데 큰 컬럼을 건드리지?

CUBRID 의 UPDATE 는 행을 **통째로 새로 쓰는** 방식입니다.
그래서 `non_oos_vc` 만 바꿔도, **바뀌지 않은 `oos_vc` 까지** 새 행에 다시 담아야 합니다.

```
UPDATE tbl SET non_oos_vc = <새 값>   (oos_vc 는 그대로 두고 싶음)
     |
     v  행 전체를 새로 만듦
   non_oos_vc - 새 값 사용 (창고 X)
   oos_vc     - "그대로" 두려면 옛 값이 필요
                -> 창고에서 꺼냄              (oos_read 1)
                -> 새 행에 넣으려고 다시 보관   (oos_insert 1)
                (* 옛 주소표 재사용 안 하고 새 주소표 발급)
```

실제 기록부(`oos.log`)에도 **꺼낸 칸(slot 1)** 과 **새로 넣은 칸(slot 0)** 이 다릅니다 -> 주소표를 재사용하지 않는다는 증거.

### 결론

> **[결론]** 작은 일반 컬럼 하나만 수정해도, 그 행에 있는 큰 OOS 컬럼은 "읽기 1 + 쓰기 1"의 비용이 듭니다.
> 오히려 **OOS 컬럼 자체를 수정하는 게 읽기 0회로 더 쌉니다** (새 값을 직접 주니까).

이건 버그가 아니라 현재(M1) 설계상 동작입니다. 값이 안 바뀌어도 항상 새 주소표를 만듭니다.
"안 바뀐 컬럼은 읽지 말고 옛 주소표를 재사용하자"는 **개선안(향후 과제, CBRD-26516 관련)** 으로 남아 있습니다.

> 규모 주의: `UPDATE tbl SET non_oos_vc = ...` 를 100개 행에 실행하면,
> 쿼리에 언급조차 안 한 `oos_vc` 때문에 **읽기 100 + 쓰기 100** 이 발생합니다.

## 8. CS 모드(클라이언트–서버)에서도 똑같은가? — 실서버 검증

위 6·7절은 `csql -S`(SA, 한 프로세스) 결과입니다. 팀에서 후속 질문이 나왔습니다:
> **실제 서버(`cub_server`)를 띄우고 클라이언트로 접속하는 CS 모드에서도 횟수가 같은가?**

결론부터: **한 개도 안 틀리고 똑같습니다.** 같은 4개 시나리오를 실서버에 대고 다시 측정했습니다.

**측정 방법 (SA 와 다른 점):**
- `cubrid server start testdb` 로 서버를 띄우고, `csql`(이번엔 `-S` **없이**)로 접속 = CS 모드
- gdb 중단점을 **`csql` 가 아니라 `cub_server` 프로세스** 에 걸고 `oos_read`/`oos_insert` 카운트
- `oos_insert` 는 `oos.log` 로 **교차 검증** (gdb 결과와 완전 일치)

| 시나리오 (행 2개) | `oos_read` (gdb) | `oos_insert` | SA 모드(6·7절) |
|---|:---:|:---:|:---:|
| `SELECT id, oos_vc` *(대조군)* | **2** | 0 | 2 / 0 ✅ |
| `SELECT id, non_oos_vc` *(팀 질문)* | **0** | 0 | 0 / 0 ✅ |
| `UPDATE SET oos_vc=… WHERE id=1` | **0** | **1** | 0 / 1 ✅ |
| `UPDATE SET non_oos_vc=… WHERE id=1` | **1** | **1** | 1 / 1 ✅ |

### 왜 똑같을 수밖에 없나

`SELECT`/`UPDATE` **문장 실행은 항상 서버 쪽** 에서 일어납니다.
쿼리 계획(XASL)이 `cub_server` 안의 스캔 매니저(`scan_manager.c`)를 타고,
`heap_attrinfo_read_dbvalues` 가 **요청한 컬럼만** 게으르게 채운 뒤,
**이미 다 풀린 값(`DB_VALUE`)** 만 클라이언트로 보냅니다.
SA 모드는 바로 그 똑같은 서버 코드를 **한 프로세스 안에서** 돌릴 뿐입니다.
→ 같은 코드, 같은 자리 = 같은 횟수.

```
[csql 클라이언트]  --SQL 문장-->  [cub_server]
                                     |
                                     v  스캔 매니저 (XASL 실행)
                          heap_attrinfo_read_dbvalues  (요청 컬럼만 resolve)
                                     |
                                     v
                          풀린 DB_VALUE 만 클라이언트로 전송
   (oos_read / oos_insert 는 전부 서버 안에서 일어남 = SA 와 동일)
```

> **주의 — 이 "동치(同値)"는 SQL 문장(SELECT/UPDATE)에 한정됩니다.**
> 클라이언트가 **레코드 한 건을 통째로 가져오는 경로**(워크스페이스 객체 fetch,
> `unloaddb`, `locator_cl` 의 인스턴스 fetch)는 **다른 길** 입니다 — 그쪽은
> `expand_oos`(CBRD-26847) 이슈가 따로 있고, 이 보고서의 측정 범위 밖입니다.

## 9. 한 장 요약

| 상황 | 창고 읽기(`oos_read`) | 창고 쓰기(`oos_insert`) |
|------|:---:|:---:|
| `SELECT id, non_oos_vc` (OOS 컬럼 조회 안 함) | **0** [O] | 0 |
| `SELECT id, oos_vc` (OOS 컬럼 조회) | 행 수만큼 | 0 |
| `UPDATE ... SET oos_vc = ...` (OOS 컬럼 수정) | **0** | 1 |
| `UPDATE ... SET non_oos_vc = ...` (일반 컬럼 수정) | **1** | 1 |

**기억할 것 3가지**
1. **조회**: 안 부른 OOS 컬럼은 절대 안 읽는다. (가장 중요, 팀 질문의 답 = YES)
2. **수정**: 행을 통째로 다시 쓰므로, 안 바꾼 OOS 컬럼도 읽고-다시쓴다. (현재 설계)
3. 숫자는 **행 개수에 비례** 한다.

> 위 4가지 횟수는 **SA 모드와 CS 모드(실서버)에서 동일** 합니다 (8절). SELECT/UPDATE 실행이
> 어차피 서버에서 도니까요. (단, 클라이언트가 레코드 통째로 fetch 하는 경로는 별도 — CBRD-26847.)

## 10. (부록) 검증하면서 만난 함정 - 다음 사람을 위해

`csql -S` 에 `gdb` 를 붙일 때 주의할 점:

1. **`csql` 는 시작할 때 자식 프로세스(`cub_pl`, 자바 저장프로시저 서버)를 하나 띄웁니다.**
   실제 SQL 엔진은 **부모 프로세스** 에서 돕니다.
   -> gdb 에서 `set follow-fork-mode parent` 로 부모를 따라가야 합니다.
   (자식을 따라가면 엉뚱한 `cub_pl` 만 보다가 끝납니다.)

2. **`oos_read` 는 성공해도 `oos.log` 에 아무것도 안 남깁니다** (실패 시에만 `oos_error` 기록).
   그래서 "로그가 비었다 = 안 읽었다"라고 **단정하면 안 됩니다.**
   이 보고서를 위해 임시로 `oos_read`/`oos_insert` 입구에 로그 한 줄을 넣어 측정했고,
   측정이 끝난 뒤 **원복(revert)** 했습니다. (소스 변경 없음)

3. 측정 도구가 멀쩡한지 **반드시 양성 대조군으로 먼저 확인** (4절 참고).

4. **CS 모드(실서버) 측정 시 추가 함정:**
   - gdb 는 **`cub_server`** 에 붙여야 합니다 (`csql` 가 아니라 — SQL 은 서버에서 돔).
     `cub_server <db>` 가 여러 개 떠 있으면 `fuser $CUBRID_DATABASES/<db>/<db>` 로
     **볼륨을 잡고 있는** 진짜 서버 PID 를 골라야 엉뚱한(죽은) 서버에 안 붙습니다.
   - gdb 를 `stdbuf -oL` 로 띄워야 중단점 hit 가 로그에 **즉시 flush** 됩니다.
     (안 그러면 블록 버퍼링 때문에 쿼리별 카운트가 어긋납니다.)
   - 이 빌드에서 `oos_insert` 는 **매 호출 로그**(oos_file.cpp:1050/1081)를 남기지만,
     `oos_read` 는 성공 시 무로그라 **읽기 카운트는 gdb 만 신뢰 가능** (2번 함정과 동일).
     실제로 처음에 `oos.log` 만 보고 "대조군조차 읽기 0" 이 나와서 측정기가 눈멀었음을
     양성 대조군이 잡아냈습니다 → gdb 로 전환.
   - (참고) ptrace 가 허용돼야 비-자식 프로세스에 attach 됩니다: `/proc/sys/kernel/yama/ptrace_scope` = 0.
