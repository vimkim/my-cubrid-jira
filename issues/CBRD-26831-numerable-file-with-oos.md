# [CBRD-26831] [OOS] [M2] [Survey] Numerable file type with OOS — bestspace tradeoff analysis

---

## TL;DR (요약)

- OOS 파일은 만들어질 때부터 **"페이지마다 번호를 매기는 파일(numerable file)"** 로 생성됩니다. 그래서 "3번째 페이지 주소 줘" 같은 요청(`file_numerable_find_nth`)을 할 수 있습니다.
- 빈 공간을 찾는 동기화 작업(bestspace sync)은 이 번호를 이용해 `1번, 2번, 3번 …` 페이지를 차례로 훑습니다. 한 번에 몇 개까지 볼지는 `clamp(10, 전체 페이지 수 × 0.2, 100)` 으로 정해져, 작은 파일이라도 최소 10개, 큰 파일이라도 최대 100개만 봅니다.
- **장점**: 첫 페이지는 고정해 두면서도 나머지 모든 페이지를 "1번부터 순서대로" 깔끔하게 훑을 수 있습니다. 복잡한 내부 비트맵을 직접 해석할 필요가 없어 코드가 단순합니다.
- **단점**:
  1. 페이지를 새로 할당할 때마다 번호표에 주소를 추가하고 그 사실을 복구 로그(WAL)에 기록해야 합니다 — 매번 드는 고정 비용입니다.
  2. 페이지를 "지움" 표시(mark-delete)한 것이 쌓이면, "n번째 페이지"를 찾는 작업이 앞에서부터 하나씩 세는 느린 방식으로 떨어집니다. 최악의 경우 동기화 전체 비용이 페이지 수에 비례(특정 경로에서는 제곱)해서 커집니다.
  3. 속도를 높여주는 캐시(`find_nth_last`)는 **임시 파일에서만** 켜지도록 잠겨 있어서, 영구 파일인 OOS 는 이 캐시를 못 씁니다.
- 즉, 지금 단계에서는 **거의 공짜로 이점만** 누리고 있지만, 나중에 페이지 삭제(vacuum)가 본격적으로 돌기 시작하면 비용과 정확성 양면에서 다시 점검해야 합니다.

## Key concepts — 먼저 알아야 할 5가지 개념

이 문서를 이해하려면 아래 다섯 단어만 알면 됩니다. 전부 쉬운 비유로 설명합니다.

| 용어 | 한 줄 뜻 | 비유 |
|---|---|---|
| **페이지 (page)** | 디스크에 데이터를 저장하는 기본 칸. 보통 16KB. | 공책의 한 "쪽". 데이터는 항상 쪽 단위로 읽고 씁니다. |
| **OOS (Out-of-row Overflow Storage)** | 너무 큰 값은 따로 보관하고, 원래 자리엔 그 위치(주소, OID 16바이트)만 남기는 저장 방식. | 도서관 카드. 카드엔 "3층 A-12 책장"이라는 위치만 적고, 두꺼운 책 자체는 책장에 둡니다. |
| **bestspace (빈 자리 찾기)** | 새 데이터를 쓸 때, 빈 새 페이지를 매번 꺼내지 않고 *이미 여유 공간이 있는 페이지* 를 빠르게 찾아주는 기능. | 주차할 때마다 새 주차장을 짓는 대신, *빈자리가 남은 기존 주차장* 을 찾는 것. 안 하면 공간 낭비가 심해집니다. |
| **numerable file (번호 매기는 파일)** | 페이지를 만든 순서대로 번호를 매겨두어, "n번째 페이지 주소 줘"라고 물어볼 수 있는 특별한 파일. | 책의 **쪽 번호 + 목차**. "152쪽 펴" 라고 바로 말할 수 있습니다. |
| **WAL (복구 로그)** | 실제로 바꾸기 *전에* "무엇을 바꿀지" 미리 일지에 적어두는 것. 컴퓨터가 갑자기 꺼져도 일지를 보고 되돌리거나 다시 적용할 수 있음. | 게임의 자동 저장. 정전돼도 마지막 저장 지점부터 복구됩니다. |

추가로 두 단어만 더:

- **sticky 페이지**: 파일의 맨 첫 페이지를 "절대 안 움직이게 고정"한 것. OOS 는 여기에 파일 전체의 통계·머리말(헤더)을 둡니다. → 책의 *표지/속표지*.
- **mark-delete (지움 표시)**: 페이지를 진짜로 즉시 지우는 대신 "이건 지운 거예요"라는 표시만 해두는 것. 진짜 청소는 나중에 vacuum(청소부)이 합니다. → 휴지통에 넣기(아직 영구삭제 전).

## Background — 왜 이 분석이 필요한가

OOS 는 "큰 컬럼 값을 별도 파일에 조각(chunk) 단위로 저장하고, 원래 레코드에는 16바이트짜리 주소(OID)만 남기는" 저장 구조입니다. 한 페이지에 다 들어가는 값은 조각 하나(single-chunk)로, 더 큰 값은 여러 조각을 사슬처럼 이어(multi-chunk) 저장합니다.

문제는 **데이터를 넣을 때마다 빈 새 페이지를 꺼내 쓰면 공간 사용률이 폭락한다** 는 점입니다(주차장을 매번 새로 짓는 셈). 그래서 같은 파일 안에서 "충분히 여유 공간이 있는 페이지"를 빠르게 찾는 **bestspace** 기능이 필요합니다.

OOS 의 bestspace 는 기존 `heap_file.c` 가 쓰던 방식을 거의 그대로 베껴 왔습니다. 머리말(헤더)에 후보 페이지 목록 `best[]` 을 두고, 전역 해시 테이블(`oos_Bestspace`)도 함께 두어, **(A) 해시 → (B) 헤더 best[] → (C) 직접 스캔(sync)** 의 3단계로 페이지를 찾습니다.

**그런데 진짜 차이는 "빈 자리 찾는 자료구조"가 아닙니다.** 진짜 차이는 *OOS 파일이 어떤 종류의 파일로 만들어졌는가* 입니다.
- heap 파일은 `file_create_heap()` 로 만들어지며 번호표(user page table)가 **없습니다.** 대신 페이지를 셀 때 "다음 페이지 주소(`next_vpid`)"를 따라가는 사슬(chain) 방식을 씁니다.
- OOS 파일은 `is_numerable=true` 로 만들어져 번호표를 **가집니다.**

이 문서는 바로 이 선택 — *"OOS 를 번호 매기는 파일로 만든 것"* — 의 장단점을 따져봅니다.

## What is a numerable file — "번호 매기는 파일"이란?

### 핵심

`file_create()` 함수의 6번째 인자 `is_numerable` 이 `true` 면, 그 파일은 **페이지를 할당한 순서대로 주소를 적어두는 목록(user page table)** 을 추가로 유지합니다. 그러면 "3번째 페이지의 실제 주소(VPID)가 뭐야?"를 `file_numerable_find_nth` 한 번으로 물어볼 수 있습니다. 책에 쪽 번호와 목차가 붙는 것과 같습니다.

> ⚠️ 한 가지 어긋난 점: 코드에는 "어떤 파일 종류가 numerable 이 될 수 있는가"를 나열한 매크로 `FILE_TYPE_CAN_BE_NUMERABLE` 이 있는데, 거기엔 확장 해시 2종과 `FILE_TEMP` 만 들어 있고 **`FILE_OOS` 는 빠져 있습니다.** 그런데 `oos_create_file` 은 이 매크로를 거치지 않고 `is_numerable=true` 를 직접 넘깁니다. 즉 "목록상 허용된 종류"와 "실제로 쓰는 곳"이 어긋나 있습니다. (지금은 문제를 일으키진 않습니다 — 아래 단점 6번 참고.)

### 일반 영구 파일과 무엇이 다른가

1. **헤더 페이지를 나누는 비율이 달라집니다.** 보통 파일은 헤더 페이지를 "빈 섹터 표 : 꽉 찬 섹터 표 = 1:1"로 씁니다. 번호 매기는 파일은 그 둘을 각각 1/32 로 줄이고, 나머지 약 30/32(16KB 기준 약 15KB)를 **번호표 저장 공간** 으로 씁니다.

2. **페이지를 할당할 때마다 번호표에 주소가 추가됩니다.** 이 일을 `file_numerable_add_page()` 가 합니다. 번호표가 한 페이지에 다 차면 추가 표 페이지(ftab page)를 더 잡아 사슬처럼 잇습니다. 그리고 영구 파일에서는 이 추가 작업이 전부 **WAL(복구 로그)로 기록** 됩니다.

3. **삭제가 두 단계로 나뉩니다.** 보통 파일은 "빈 칸 표시 비트만 끄면" 끝입니다. 번호 매기는 파일은 거기에 더해 번호표의 해당 자리에 **"지움 표시(mark-delete)"** 를 세팅하고, 그 사실도 로그로 남겨야 합니다.

4. **이름 그대로 "셀 수 있다(numerable)"** — `file_numerable_find_nth` 로 n번째 페이지를 바로 찾는 게 핵심 기능입니다.

### `file_numerable_find_nth` 의 두 가지 동작 모드 — 여기가 비용의 핵심

- **지움 표시가 하나도 없을 때 (빠름)**: 번호표를 덩어리(component) 단위로 건너뛰며 점프해서 들어갑니다. 빠릅니다.
- **지움 표시가 쌓였을 때 (느림)**: 앞에서부터 칸을 하나씩 보면서, 지움 표시된 칸은 건너뛰고 살아있는 칸을 만날 때마다 "남은 카운트"를 1씩 줄이다가 0이 되면 멈춥니다. 즉 한 번 호출에 **번호표 전체 길이(n)만큼 걸을 수도 있습니다.** (한 번 호출 최악 = O(n))

> 비유: 책 목차가 멀쩡하면 "152쪽" 바로 펼치지만, 목차의 절반이 찢겨(지움 표시) 있으면 첫 쪽부터 한 장씩 넘기며 세야 하는 상황.

### 속도 캐시가 있지만… OOS 엔 안 켜집니다

헤더에는 "직전에 찾은 위치"를 기억해 다음 검색을 빠르게 해주는 캐시(`find_nth_last`)가 있습니다. 그런데 이 캐시가 켜지는 조건은 다음 **4개가 전부 동시에** 참이어야 합니다.

```
numerable 파일?  AND  임시 파일?  AND  종류가 FILE_TEMP?  AND  병렬 스레드 아님?
```

OOS 는 **영구 파일** 이고 종류도 `FILE_OOS` 이므로, "임시 파일?" 과 "FILE_TEMP?" 두 조건에서 막혀 **캐시를 못 씁니다.** 그래서 동기화가 `1번, 2번, 3번 …` 으로 부를 때마다 매번 번호표 처음부터 다시 셉니다.

## OOS bestspace use case — OOS 는 이 기능을 어디에 쓰나

`oos_find_best_page` 는 4단계로 후보 페이지를 정합니다.
1. 고정된 첫 페이지(헤더)를 잡고, 헤더의 `best[]` 목록과 전역 해시를 함께 봅니다.
2. `oos_stats_find_page_in_bestspace` 가 **(A) 전역 해시 → (B) 헤더 best[] → (C) 후보 페이지를 살짝 잠가보고 실제 빈 공간 확인** 순서로 시도합니다. 이때 캐시는 단순 조회가 아니라 **"조회하면서 안 맞으면 그 자리에서 캐시에서 빼버리는(evict)"** 방식입니다.
3. 다 실패하고 "여유 페이지 비율"이 임계치(`0.1`) 이상이면, 헤더 잠금을 잠깐 풀고 **직접 스캔(sync)** 을 합니다. ← **번호 매기는 파일 기능이 등장하는 곳이 바로 여기입니다.**
4. 그래도 없으면 새 페이지를 새로 할당합니다.

직접 스캔(`oos_stats_sync_bestspace`)은 전체 페이지 수를 구한 뒤 `start_idx = 1`(0번=헤더는 건너뜀)부터 시작해서, 단 한 줄로 페이지를 순회합니다:

```c
for (int i = 1; i < total_pages && iterations < max_iterations; i++)
    file_numerable_find_nth (..., i, ...);   // i번째 페이지 주소를 받아온다
```

만약 번호 기능이 없었다면, OOS 가 (i) heap 처럼 페이지마다 "다음 페이지 주소"를 박아 사슬을 따라 돌거나, (ii) 내부 섹터 비트맵을 직접 풀어 페이지를 나열하는 코드를 **직접 짜야** 했을 겁니다.

## Pros — 장점

1. **동기화 코드가 단순합니다.** 위에서 본 것처럼 페이지 순회가 사실상 한 줄(`for` + `file_numerable_find_nth`)로 끝납니다. heap 쪽 같은 일을 하는 `heap_stats_sync_bestspace` 는 `next_vpid` 사슬을 따라가느라 코드가 훨씬 깁니다. (실측 비교는 없는 정성적 판단입니다.)

2. **헤더 페이지가 자연스럽게 0번이 됩니다.** 고정 첫 페이지가 번호표 0번 자리가 되므로, 동기화는 `start_idx = 1` 로 안전하게 헤더를 건너뜁니다. 게다가 해시·best[] 후보에서도 헤더가 안 잡히도록 한 번 더 걸러내는 방어선이 있습니다.

3. **"어디까지 봤는지" 이어보기가 쉽습니다.** 동기화는 `clamp(10, 전체×0.2, 100)` 만큼만 보고 멈춘 뒤 재개 지점을 저장합니다. 번호가 있으면 "n번까지 봤다"가 그냥 정수 하나로 표현됩니다. (다만 현재는 코드에 `/* TODO: ideally find the index of full_search_vpid; for now start from 1 */` 라고 적혀 있고, 매번 1부터 다시 시작합니다.)

### 아직 실현되지 않은 미래 이점

- `file_numerable_truncate` 라는 형제 API 가 이미 존재합니다. "n번째 이후 페이지를 전부 삭제"하는 기능이라, 나중에 OOS 파일을 줄이거나(shrink) 정리할 때 따로 만들 필요 없이 쓸 수 있습니다. 다만 **지금은 아무도 호출하지 않으므로** 장점이 아니라 *잠재적 미래 이점* 으로만 둡니다.

---

## Cons — 단점

> 아래 항목은 전부 **코드 구조와 알고리즘 비용을 따진 분석** 이며, 실제 성능 측정(벤치마크, 프로파일)은 아직 없습니다. 정량 판단이 필요한 결정은 따로 측정해야 합니다.

1. **페이지를 새로 할당할 때마다 "번호표 추가 + WAL"이라는 고정 비용을 더 냅니다.** OOS 페이로드 자체의 로그(`RVOOS_INSERT`)와는 **별개** 로 듭니다. 번호표가 가득 차서 표 페이지(ftab)를 새로 잡을 때도 별도 로그가 또 찍힙니다. (정확한 바이트 크기는 측정 안 함.)

2. **지움 표시가 쌓이면 동기화 최악 비용이 페이지 수에 비례합니다.** 지움 표시가 있으면 `file_numerable_find_nth` 가 "앞에서부터 하나씩 세는" 느린 경로로 빠집니다(한 호출 최악 O(n)). 동기화는 `i = 1, 2, …, max_iterations` 로 반복하므로 전체 최악 비용은 **`max_iterations × n`** 에 비례합니다. `scan_all=true` 인 경로에서는 `max_iterations = total_pages` 가 되어 **`n²`(제곱)** 에 비례합니다.
   > 즉, 페이지가 2배가 되면 이 경로의 최악 시간은 약 4배가 될 수 있다는 뜻입니다.

   지금은 `oos_remove_page` 가 아직 TODO(미연동) 라서 지움 표시가 거의 안 쌓이지만, vacuum 이 연동되면 이 경로로 지움 표시가 누적되기 시작합니다.

3. **속도 캐시가 OOS 에선 꺼져 있습니다.** 위에서 설명한 4중 조건 때문에, 영구 파일인 OOS 는 `find_nth_last` 캐시를 못 씁니다. 그래서 동기화가 매번 번호표 처음부터 다시 인덱싱합니다. (얄궂게도 OOS 의 호출 패턴 `i=1,2,3,…` 은 캐시가 가장 잘 들을 단조 증가 패턴인데도 게이트에 막혀 있습니다.)

4. **OOS 와 heap 이 서로 다른 방식으로 페이지를 나열합니다.** heap 은 페이지마다 "다음 페이지 주소(`next_vpid`)" 사슬을 따라가고, OOS 는 파일 차원의 번호표를 씁니다. 같은 목적(페이지 한 바퀴 훑기)을 두 모듈이 다른 메커니즘으로 풀고 있어서, OOS 는 *코드 단순함* 을 얻는 대신 *번호표 유지 비용*(헤더 분할 변화, 매 할당마다 추가+WAL, 지움 표시 복구 처리)을 떠안았습니다.

5. **헤더 페이지가 "단일 고장점"이 됩니다.** 번호표가 헤더의 약 15/16 을 차지하고, OOS 통계(`OOS_HDR_STATS`, 약 200바이트)도 같은 헤더 페이지에 들어갑니다. 둘이 같은 페이지의 다른 영역에 공존하다 보니, 동기화 중에 헤더 잠금을 잠깐 풀었다 다시 잡는 별도 처리가 필요해졌습니다. 전역 해시 무효화 경로까지 더해지면, 헤더와 전역 해시가 동시에 "여기 막히면 다 막히는" 지점이 됩니다.

6. **`FILE_TYPE_CAN_BE_NUMERABLE` 매크로의 일관성이 깨져 있습니다.** 매크로엔 `FILE_OOS` 가 없는데 OOS 는 우회해서 numerable 로 만듭니다. 다행히 이 매크로가 실제로 쓰이는 단 한 곳(`file_dealloc` 의 조기 종료 가드)에서는, `FILE_OOS` 면 그냥 early-exit 하고 그 직후 별도의 `FILE_IS_NUMERABLE` 체크가 한 번 더 막아주므로 **지금은 사고가 안 납니다.** 다만 이 매크로가 다른 곳에서 "불변식(invariant)"으로 쓰이기 시작하면 OOS 가 누락될 위험이 있습니다.

7. **복구(recovery)에서 신경 쓸 표면이 넓어집니다.** 번호표 자체의 redo/undo 로그(`RVFL_USER_PAGE_MARK_DELETE`, `RVFL_FHEAD_SET_LAST_USER_PAGE_FTAB`, `file_log_extdata_add`)가 추가로 활성화됩니다. OOS 조각 단위 로그(`RVOOS_INSERT/DELETE`)와는 **별개 경로** 라, 크래시 복구 시 두 경로가 서로 모순 없이 잘 맞물리는지(예: 번호표 추가 redo 와 조각 삽입 redo 의 순서) 새로 검증해야 합니다. (이 정확성 분석은 본 문서 범위 밖.)

## Suggestions for improvements — 개선 제안

> 우선순위는 "지금 당장 vs. vacuum 연동 이후"를 기준으로 매겼습니다.

### 지금(M2) 해도 되는 것 — 저비용·저위험

- **S1. 캐시 게이트를 영구 OOS 까지 확장 검토 (가장 효과 클 가능성).**
  `FILE_CACHE_LAST_FIND_NTH` 의 4중 조건 중 "임시 파일 / FILE_TEMP" 두 개가 OOS 를 막고 있습니다. OOS 의 호출 패턴(`i=1,2,3,…` 단조 증가)은 external sort 가 캐시를 쓰는 패턴과 **사실상 동일** 하므로, 게이트를 permanent OOS 까지 넓히면 동기화의 "매번 처음부터 다시 인덱싱"을 없앨 수 있습니다.
  - ⚠️ 단, 게이트에는 `m_px_orig_thread_entry == NULL`(병렬 스레드가 아니어야 함) 조건도 있습니다. OOS 의 동시 접근 시나리오와 충돌하지 않는지 **먼저 검증** 해야 합니다. 캐시는 단방향(직전 위치)이라, 동시에 두 스레드가 같은 파일을 nth 로 훑으면 캐시가 오염될 수 있기 때문입니다.

- **S2. `full_search_vpid` 재개를 진짜 인덱스 기반으로.**
  지금은 코드 주석대로 매 라운드 `start_idx = 1` 로 되돌아가 앞부분을 다시 봅니다(`/* TODO … for now start from 1 */`). 저장해 둔 재개 VPID 의 인덱스를 한 번 구해 거기서부터 이어보면, 같은 앞쪽 페이지를 반복해서 훑는 낭비를 줄일 수 있습니다. S1 과 합치면 "이어보기"가 캐시 덕에 더 싸집니다.

- **S3. `FILE_TYPE_CAN_BE_NUMERABLE` 매크로는 *지금은 건드리지 말 것* (의도적 보류).**
  이 매크로의 유일한 사용처가 `file_dealloc` 의 조기 종료 가드라서, 여기에 `FILE_OOS` 를 추가하면 `file_dealloc(FILE_OOS, …)` 가 early-exit 대신 **번호표 지움-표시 분기로 진입** 하게 됩니다. 즉 동작이 바뀝니다. 그러므로 매크로 갱신은 `oos_remove_page` 가 실제로 활성화되는 시점(아래 S4)과 **함께** 해야 안전합니다. 지금 당장은 주석으로 "OOS 는 의도적으로 우회 중"임을 명시하는 정도가 적절합니다.

- **S4. 성능 회귀를 잡을 측정 훅을 미리 심기.**
  현재 단점들은 전부 "분석상 그렇다"일 뿐 실측이 없습니다. `oos_stats_sync_bestspace` 의 호출 횟수·순회한 엔트리 수·소요 시간을 통계로 남겨두면, vacuum 연동 후 회귀를 **즉시** 포착할 수 있습니다(추측이 아니라 숫자로).

### vacuum 연동(M3 이후) 시점에 결정할 것 — 고비용·고영향

- **S5. 지움 표시가 쌓이는 환경에서 동기화 성능을 실측하고, 필요하면 numerable 의존을 끊기.**
  `oos_remove_page` 가 실제로 페이지를 삭제하기 시작하면 단점 2번(`max_iterations × n`, 최악 `n²`)이 현실화됩니다. 측정 결과 회귀가 확인되면, 동기화를 다음 중 하나로 다시 쓰는 것을 검토할 만합니다.
  - **(a) heap 방식**: OOS 페이지 헤더에 `next_vpid` 사슬을 박아 따라 돌기. → 번호표 비용(헤더 분할·매 할당 WAL·지움 표시 undo)을 통째로 없애는 대신, 페이지마다 사슬 무결성을 위한 update WAL 을 새로 들임.
  - **(b) 섹터 비트맵 방식**: 이미 있는 `file_get_all_data_sectors` 헬퍼로 빈/꽉 찬 섹터를 풀어 페이지를 직접 나열. external sort 가 쓰는 방식과 같은 계열.
  - 이 둘은 단점 4번이 가리키는 trade-off 의 *반대쪽 끝* 입니다. 측정 없이 미리 바꾸지 말고, **S4 의 숫자가 나온 뒤** 결정해야 합니다.

- **S6. 두 복구 경로의 상호작용을 회귀 테스트로 고정.**
  번호표 로그(redo/undo)와 OOS 조각 로그가 별개 경로이므로(단점 7번), 크래시-복구 시나리오 테스트(특히 번호표 ftab 확장 직후 크래시, 지움 표시 직후 크래시)를 추가해 idempotency 와 순서를 못 박아 두는 것이 좋습니다.

---

## Conclusion — 결론

OOS 를 `is_numerable=true` 로 만든 결정은, **동기화 스캔에서 페이지 순회를 한 줄로 끝낼 수 있게 해준** 대신 다음 비용을 떠안았습니다.

- (a) 페이지를 할당할 때마다 "번호표 추가 + WAL"이라는 고정 비용,
- (b) 지움 표시가 쌓이는 환경에서 동기화의 `max_iterations × n`(특정 경로 `n²`) 최악 비용,
- (c) 영구 파일이라 `find_nth_last` 속도 캐시를 못 쓰는 점,
- (d) 번호표 관련 복구 경로가 늘어나는 점.

현재 마일스톤(M2 — bestspace 도입)까지는 `file_dealloc`(=페이지 삭제)이 거의 호출되지 않는다는 가정 아래, **numerable 의 이점만 거의 공짜로 누리는 상태** 로 보입니다(`oos_remove_page` 는 vacuum 연동 시점까지 TODO 보류).

따라서 권장 순서는 명확합니다.
1. **지금**: S4(측정 훅)를 심고, S1·S2(캐시·재개)를 *검증과 함께* 검토하며, S3(매크로 보류)을 지킨다.
2. **vacuum 연동 후**: S4 가 뽑아낸 숫자를 보고, 회귀가 있으면 S5(heap 사슬 / 섹터 비트맵으로 재작성)와 S6(복구 회귀 테스트)을 결정한다.

핵심은 **"추측으로 미리 갈아엎지 말고, vacuum 이 켜진 뒤 실측 숫자로 결정한다"** 입니다.
