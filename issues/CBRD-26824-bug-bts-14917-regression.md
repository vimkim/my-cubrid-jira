# [OOS] bug_bts_14917 테스트가 점점 느려지다 시간 초과로 실패하던 문제

> 이 문서는 고등학생도 이해할 수 있도록 쉬운 말로 다시 썼습니다.
> 코드 수준의 자세한 내용은 맨 아래 **"엔지니어용 기술 참고"** 에 따로 모아 두었습니다.

## 한 줄 요약

큰 사진(약 3MB) 을 데이터베이스에 계속 집어넣는 테스트가 있는데, **집어넣을수록 한 장 넣는 시간이 점점 길어져서** 결국 정해진 제한 시간(720초) 안에 끝나지 못하고 실패했습니다.
원인은 "빈 자리를 찾는 코드" 가 **빈 자리가 없는 게 뻔한 상황에서도 매번 창고 전체를 헛되이 훑어보고 있었던 것** 이었고, 그 헛수고를 안 하도록 한 줄을 고쳐서 해결했습니다. (10번 반복 기준 마지막 한 장이 **890ms → 71ms, 약 12배** 빨라짐)

---

## 먼저, 배경 지식 몇 가지 (쉬운 말로)

데이터베이스는 표(table) 에 데이터를 저장합니다. 표의 한 줄(row) 은 **"페이지(page)"** 라는 정해진 크기의 칸(약 16KB) 에 들어갑니다. 작은 데이터는 한 칸에 쏙 들어갑니다.

그런데 **사진처럼 아주 큰 데이터(3MB)** 는 한 칸에 도저히 안 들어갑니다. 그래서 CUBRID 에는 **OOS(Out-of-row Storage)** 라는 기능이 있습니다.

- **OOS = 큰 데이터를 따로 빼서 보관하는 별도 창고.**
  옷이 옷장에 안 들어가면 따로 창고에 두는 것과 똑같습니다. 큰 컬럼만 떼어내서 "OOS 파일" 이라는 별도 공간에 저장합니다.
- **조각(chunk)**: 3MB 짜리도 창고의 칸 하나(16KB) 에는 한 번에 안 들어갑니다. 그래서 **약 200 조각으로 쪼개서 200 개 칸에 나눠 담습니다.** 이 조각 하나하나를 chunk 라고 부릅니다.
- **빈 자리 찾기**: 새 조각을 넣을 때, 데이터베이스는 공간 낭비를 줄이려고 **"아직 자리가 남은 칸"** 을 먼저 찾아서 거기에 채워 넣으려 합니다. 이를 위해 "빈 자리 목록(메모장)" 을 봅니다. 목록에 마땅한 칸이 없으면, **창고를 직접 처음부터 훑어보며** 빈 칸을 찾습니다. (이 훑어보는 동작을 sync scan 이라고 부릅니다.)

## 무슨 일이 있었나 (증상)

테스트 이름은 `bug_bts_14917` 입니다. 하는 일은 간단합니다.

```
100번 반복:
  3MB 짜리 사진 20장을 표에 INSERT(집어넣기)
  방금 넣은 것들을 DELETE(지우기)
```

이 테스트는 두 가지 문제로 무너졌습니다.

1. **1단계 — 엉뚱한 오류 메시지 (이미 해결됨)**
   사진을 읽는 도중 "인덱스가 깨졌다" 는 가짜 오류가 떠서 테스트가 약 18초 만에 죽었습니다.
   → 이건 다른 티켓 **CBRD-26815** 에서 이미 고쳤습니다.

2. **2단계 — 점점 느려짐 (이번에 해결한 문제)**
   사진 한 장 넣는 시간이 처음엔 빠른데 **갈수록 느려졌습니다.** 반복할수록 약 3.5배까지 느려져서, 100번을 720초 안에 끝낼 수 없었습니다.

**영향**: 이 테스트 하나가 실패하면 CI(자동 검사) 가 빨간불이 되고, 다른 PR 들의 통과까지 막혔습니다. (데이터가 망가지는 문제는 아니었고, 순전히 "속도" 문제였습니다.)

---

## 왜 점점 느려졌나 (핵심 원인)

비유로 설명하면 이렇습니다.

> 우리 데이터는 **조각 하나가 창고의 칸 하나를 거의 꽉 채웁니다.**
> 그러면 "자리가 조금 남은 칸" 이 있어도 **새 조각이 거기에 안 들어갑니다** (조각이 너무 커서).
> 즉, 빈 자리 찾기는 **항상 헛수고** 입니다. 어차피 새 칸을 하나 더 만들어야 합니다.

그런데 코드가 이렇게 되어 있었습니다.

> "일단 **무조건 한 번은** 창고를 처음부터 훑어봐라."

그래서 **조각을 넣을 때마다(사진 한 장당 약 200번)** 창고를 헛되이 훑었습니다.
창고(OOS 파일) 가 커질수록 훑는 데 시간이 더 걸리니까, **넣으면 넣을수록 한 장당 시간이 길어진** 것입니다.

**결정적 단서**: CUBRID 에는 이미 비슷한 기존 코드(heap 의 빈 자리 찾기) 가 있는데, 그 코드는 훨씬 똑똑합니다.

> "**빈 칸이 어딘가 있다는 신호가 있을 때만** 창고를 훑어라. 신호가 없으면 그냥 새 칸을 만들어라."

OOS 코드는 이 기존 코드를 베껴 만들면서, 실수로 **"무조건 한 번 훑어라"** 라는 조건을 덧붙였습니다. 바로 이 한 줄이 원인이었습니다.

### 측정으로 확인한 느려짐 (수정 전)

사진 한 장(=20줄 중 1줄) 넣는 데 걸린 시간:

| 반복 횟수 | 한 장당 시간 |
|---:|---:|
| 0번째 | 약 164 ms |
| 5번째 | 약 613 ms |
| 9번째 | 약 890 ms (처음의 5배 이상) |

그래프가 계속 **우상향(점점 위로)** 했습니다. 이게 "창고가 커질수록 비싸지는 동작이 있다" 는 증거입니다.

---

## 어떻게 고쳤나

`oos_find_best_page` 라는 함수에서, **"무조건 한 번 훑어라"** 에 해당하는 조건 한 줄을 지웠습니다.

```diff
-  if (빈자리비율 >= 기준값 || 첫_시도라면)   // ← "첫 시도면 무조건 훑어라" 부분이 문제
+  if (빈자리비율 >= 기준값)                   // ← 빈 칸 신호가 있을 때만 훑는다 (기존 heap 코드와 동일)
```

이렇게 하면 OOS 도 기존 heap 코드와 **똑같이** 동작합니다. "빈 칸이 있다는 신호가 있을 때만 훑고, 없으면 바로 새 칸을 만든다."

**걱정되는 점도 확인했습니다.** "지운 자리를 다시 쓰는(재활용) 기능이 망가지지 않나?"
→ 괜찮습니다. 재활용은 **항상 최신으로 갱신되는 빈 자리 목록(메모장)** 을 통해 처리되고, 이번에 손댄 "창고 훑기" 는 그 메모장이 비었을 때만 쓰는 보조 수단이기 때문입니다. (이를 검증하는 단위 테스트도 통과합니다.)

## 남은 이야기 / 주의할 점

- **이번 수정은 2단계(느려짐) 를 해결합니다.** 티켓에서 가장 유력한 원인으로 지목했던 "창고 훑기 빈도" 가 실제 범인이었습니다.
- 수정 후에도 아주 약간 느려지는 부분은 남아 있습니다. 이건 OOS 만의 문제가 아니라, **데이터를 GB 단위로 계속 쓰면 생기는 일반적인 디스크/메모리 비용**(새 칸 할당, 메모리 캐시 교체) 입니다. 규모가 수백 ms → 수십 ms 수준으로 **훨씬 작아졌습니다.** 필요하면 따로 추적합니다.
- **1단계(가짜 오류) 수정인 CBRD-26815 는 이 작업 브랜치에 아직 없습니다.** 그래서 `bug_bts_14917` 테스트가 처음부터 끝까지 완전히 통과하려면 CBRD-26815 도 함께 합쳐져야 합니다.

## 엔지니어용 기술 참고

> 아래부터는 코드를 직접 보는 사람을 위한 요약입니다. 위 내용으로 이해에 충분하면 안 봐도 됩니다.

### 근본 원인

`oos_find_best_page` (`src/storage/oos_file.cpp`) 의 best-space sync 진입 조건:

```c
// 수정 전
if (ratio >= OOS_BESTSPACE_SYNC_THRESHOLD || try_find == 0)
// 수정 후
if (ratio >= OOS_BESTSPACE_SYNC_THRESHOLD)
```

- `ratio = num_other_high_best / num_pages` (파일 어딘가에 여유 page 가 있다는 추정치).
- 원래 heap 의 `heap_stats_find_best_page` (`src/storage/heap_file.c`) 는 이 ratio 가 threshold 이상일 때만 `*_sync_bestspace` 스캔을 돈다. "we will find free pages ... only if we know that a free page exists somewhere." 라는 주석 그대로다.
- OOS 는 그 로직을 포팅하면서 `|| try_find == 0` 을 덧붙여, **매 INSERT 첫 시도마다 무조건 sync 스캔** 을 강제했다.
- blob workload 에서는 chunk 하나가 page 를 거의 꽉 채우므로 부분 여유 page 가 chunk 를 못 받는다 → `num_other_high_best` 가 0 근처로 유지됨 → 그런데도 chunk 마다 sync 스캔(`min(total_pages*0.2, 100)` page 를 `pgbuf_fix`) 이 헛돌고, 그 비용이 파일 크기에 비례해 커졌다.

### 동작 흐름 (hot path)

```
oos_insert
└── oos_insert_across_pages       // 3MB → 약 200 chunk
    └── (chunk 마다) oos_insert_within_page
        └── oos_find_best_page
            ├── 1) global hash cache + 헤더 best[] 힌트 조회
            ├── 2) 실패 시 oos_stats_sync_bestspace  ← 여기가 헛돌던 지점
            └── 3) 그래도 없으면 새 page 할당
```

### 핵심 파일

| 위치 | 설명 |
|---|---|
| `src/storage/oos_file.cpp` `oos_find_best_page` | 이번 수정 지점 (sync 진입 gate) |
| `src/storage/heap_file.c` `heap_stats_find_best_page` | 그대로 맞춰야 할 기존 heap 로직 |

### 검증 환경

- standalone(SA) 모드 probe, 3MB BIT VARYING blob, 20 row/iter, 10 iter.
- 수정 전 164→890 ms (5.4배), 수정 후 20→71 ms (iter 9 기준 12.5배 개선).
- 데이터 정합성: 200 blob 전부 원본과 byte-identical (3,145,728 byte).
- OOS unit test 14/14 통과.

- end-to-end shell test 완전 통과에는 1단계 fix(CBRD-26815, `skip_oos_expand`) 동반 머지 필요.
