## Findings

### F1. [NUMERIC] 비교 경로 auto-cast precision 미적용 — CBRD-26727

**배경**: CBRD-26652 가 문자 리터럴 → NUMERIC auto-cast default precision 을 `DB_DEFAULT_NUMERIC_PRECISION`(15) → `DB_MAX_NUMERIC_PRECISION`(38) 로 확장. `src/parser/type_checking.c:4815` 한 줄 수정(`pt_coerce_expression_argument`). 그러나 **산술 경로에만 적용** 되고 **비교/BETWEEN/CASE/UPDATE WHERE** 경로에는 미반영.

**재현**:
```sql
CREATE TABLE tnum(v NUMERIC(38,9)); INSERT INTO tnum VALUES (1234567);

SELECT COUNT(*) FROM tnum WHERE v > '9999999';                   -- ERROR 7자리
SELECT COUNT(*) FROM tnum WHERE v BETWEEN '1' AND '9999999';     -- ERROR
SELECT COUNT(*) FROM tnum WHERE v = '1234567';                   -- ERROR
SELECT COUNT(*) FROM tnum WHERE v != '1234567';                  -- ERROR
SELECT COUNT(*) FROM tnum WHERE v <= '1234568';                  -- ERROR
SELECT CASE WHEN v > '1000000' THEN 'b' ELSE 's' END FROM tnum;  -- ERROR
UPDATE tnum SET v = v WHERE v > '1000000';                       -- ERROR

SELECT COUNT(*) FROM tnum WHERE v IN ('1000000','1234567');      -- OK
SELECT v + '9999999'  FROM tnum;                                 -- OK
SELECT v > 9999999   FROM tnum;                                  -- (integer literal) OK
```

**경계값**: 정확히 **정수부 6자리 (`'999999'`)** 까지만 비교 경로가 동작. 이는 기존 `NUMERIC(15, 9)`(정수부 15−9=6) 그대로.

**의심 위치**: `pt_wrap_with_cast_op`, `tp_value_coerce`, 비교 operator 경로의 string→numeric coercion.

**자세한 이슈**: `/home/vimkim/gh/my-cubrid-jira/issues/CBRD-26727-numeric-compare-coerce.md`

### F2. [PARSER] 단일 뷰 머지 시 ORDERED/LEADING 힌트 정제 미동작 — CBRD-26728

**배경**: CBRD-26419 의 spec 은 (rule 2) 주질의에 LEADING 이 있으면 부질의 LEADING 은 머지된 질의로 복사하지 않는다, (rule 3) 부질의의 ORDERED 힌트는 뷰 머지 시 제거한다. 11.4.5 빌드에서 `SHOW TRACE` 로 rewritten query 를 확인하면 **단일 뷰 머지** 경로에서 두 규칙이 전혀 동작하지 않음.

**재현**:
```sql
SET TRACE ON;
-- Case A: ORDERED 제거되어야 함
SELECT /*+ recompile */ *
FROM ( SELECT /*+ ORDERED */ a.v FROM t1 a, t2 b WHERE a.id = b.id ) v;
SHOW TRACE;
-- rewritten: select /*+ ORDERED */ v.v ...   ← ORDERED 남음 (제거되어야 함)

-- Case B: outer LEADING 이 있으면 inner LEADING 은 복사되지 않아야 함
SELECT /*+ recompile LEADING(V) */ *
FROM ( SELECT /*+ LEADING(b, a) */ a.v FROM t1 a, t2 b WHERE a.id = b.id ) V;
SHOW TRACE;
-- rewritten: select /*+ LEADING(V, b, a) */ ...   ← inner LEADING append 됨 (복사되면 안 됨)

-- Case C (baseline): outer LEADING 없고 inner LEADING 만 있으면 복사 OK
SELECT /*+ recompile */ *
FROM ( SELECT /*+ LEADING(b, a) */ a.v FROM t1 a, t2 b WHERE a.id = b.id ) V;
-- rewritten: select /*+ LEADING(b, a) */ ...   ← OK (의도대로)
```

**근본 원인**: `src/parser/view_transform.c:14702` `mq_copy_sql_hint()`

```c
/* remove some hints if there are multiple tables.  */
if (dest_query->info.query.q.select.from->next != NULL)  // ← gate
  {
    /* ignore ordered hint */
    if (src_query->info.query.q.select.hint & PT_HINT_ORDERED) ...
    if (src_query->info.query.q.select.hint & PT_HINT_LEADING
        && dest_query->info.query.q.select.hint & PT_HINT_LEADING) ...
  }
```

이 함수는 호출 사이트(`view_transform.c:2597/2798/2838`)에서 **치환 전** 에 실행된다. 단일 뷰 머지 시 `dest->from` 은 view-spec 한 개뿐 → `from->next == NULL` → gate false → 정제 블록 skip → 이어지는 `parser_append_node` 가 무조건 LEADING 을 append.

**자세한 이슈**: `/home/vimkim/gh/my-cubrid-jira/issues/CBRD-26728-viewmerge-hint-gate.md`

추가 관측:
- Case D: outer `/*+ LEADING(V) */` + inner `/*+ ORDERED */` → rewritten 은 `/*+ ORDERED */ V.v from t1 V, t2 b` (LEADING 이 누락되고 ORDERED 만 남음) — 또 다른 정제 실패 패턴
- Case E: outer `/*+ NO_MERGE */` (인자 없음) 에도 불구하고 Query Plan 이 2-테이블 flat scan 으로 rewrite → NO_MERGE 가 무시되는 것처럼 보임. 인자 없는 NO_MERGE 의 의미 확인 필요 (F5 와 중첩 가능).

### F3. [DDL] `DROP IF EXISTS <synonym-name>` (타입 키워드 없음) 이 조용히 no-op

**배경**: CBRD-26541 이전에는 `DROP IF EXISTS syn_name` 이 **backing table 을 drop 하는** 버그가 있었다. 11.4.5 에서는 backing table 은 보존된다 (fix OK). 그러나 **시노님 자체도 삭제되지 않고** `Execute OK` 로 조용히 종료된다.

**재현**:
```sql
CREATE TABLE real_t(id INT);
CREATE SYNONYM syn_for_real_t FOR real_t;
DROP IF EXISTS syn_for_real_t;                                  -- Execute OK
SELECT synonym_name FROM db_synonym WHERE synonym_name='syn_for_real_t';
-- → 'syn_for_real_t' (아직 존재)
```

**권고** (택 1):
1. `DROP IF EXISTS <name>` 에 타입 키워드가 없을 때 시노님을 우선 탐색하여 drop
2. 명확히 "cannot determine object type" 오류 발생
3. 현재 동작을 유지하되 문서에 "타입 키워드 필수" 명시

**Impact**: 고객/자동화 스크립트가 DROP 성공으로 오인, 이후 `CREATE SYNONYM` 이 "already exists" 에러로 실패.

### F4. [DDL] `DROP PROCEDURE / FUNCTION IF EXISTS` 가 syntax error

**재현**:
```sql
DROP TABLE     IF EXISTS t;       -- OK
DROP VIEW      IF EXISTS v;       -- OK
DROP SYNONYM   IF EXISTS s;       -- OK
DROP PROCEDURE IF EXISTS p;       -- ERROR: Syntax error: unexpected 'IF'
DROP FUNCTION  IF EXISTS f;       -- ERROR: Syntax error: unexpected 'IF'
```

**Impact**: PL/CSQL 관련 스크립트를 idempotent 하게 작성하기 위해 `CREATE OR REPLACE` 는 가능하나, 관리형 드롭이 어려움. MySQL/PostgreSQL/Oracle 모두 `DROP PROCEDURE IF EXISTS` 를 지원.

**권고**: parser 의 `drop_stmt` / `drop_procedure_stmt` rule 에 `IF EXISTS` 옵션 추가. 실패 시 warning 으로만 처리.

### F5. [HINT] `/*+ NO_MERGE */` 인자 없이 쓰인 경우 동작 불분명

**관측**: F2 의 Case E 에서 `SELECT /*+ recompile NO_MERGE */ * FROM (SELECT /*+ ORDERED */ ... ) V;` 를 실행했을 때 `SHOW TRACE` 의 rewritten query 가 이미 view-merged 된 형태(`from [dba.t1] V, [dba.t2] b`). CUBRID 매뉴얼의 NO_MERGE 사용법과 인자 없음 케이스의 기대 동작 확인 필요.

**제안**: 구문 검증 + 문서 확인 후 (a) NO_MERGE 무인자 허용 & 전체 뷰 머지 차단, 또는 (b) parser 수준에서 에러로 처리.

### F7. [OPTIMIZER] View merge 시 outer `USE_MERGE` 힌트 silent drop — CBRD-26735

**배경**: View merge 경로에서 outer SELECT 의 `/*+ USE_MERGE(view_alias, other_table) */` 힌트가 통째로 사라져 MERGE JOIN 이 NL 로 폴백된다. 동일 구조에서 `USE_NL` 과 `USE_HASH` 는 정상 보존되므로 `USE_MERGE` 한 종류만 선별적으로 깨져 있다.

**재현**:
```sql
CREATE TABLE t1(id INT PRIMARY KEY, v INT);
CREATE TABLE t2(id INT PRIMARY KEY, v INT);
INSERT INTO t1 SELECT ROWNUM, ROWNUM*2 FROM db_class LIMIT 50;
INSERT INTO t2 SELECT ROWNUM, ROWNUM*3 FROM db_class LIMIT 50;

SET TRACE ON;
SELECT /*+ RECOMPILE USE_MERGE(v,b) */ *
FROM (SELECT a.v FROM t1 a, t2 b WHERE a.id=b.id) v;
SHOW TRACE;
-- rewritten: select v.v from [dba.t1] v, [dba.t2] b where (v.id=b.id)
--            ↑ USE_MERGE(v,b) 완전히 사라짐, 실행은 NL
```

**Counter-example** (같은 구조에서 USE_NL / USE_HASH 는 정상):
```sql
SELECT /*+ RECOMPILE USE_NL(v,b)   */ * FROM (...) v;  -- USE_NL 살아남음
SELECT /*+ RECOMPILE USE_HASH(v,b) */ * FROM (...) v;  -- USE_HASH 살아남음
```

**Root cause**: `src/parser/view_transform.c:14749` `mq_copy_sql_hint()` 의 `use_merge` 분기가 subquery 노드 리스트를 무조건 append 하지만, outer 의 기존 `use_merge` arg 노드들의 `spec_id` 가 view-spec 치환 후 stale 상태로 남음. `MQ_FIX_SPEC_ID` 순회(line 1776)는 새로 append 된 subquery 노드만 다루고 outer 원본은 미순회 → printer 에서 silent drop. `PT_HINT_LEADING` 경로(14709–14714)에 있는 conflict guard 가 `use_merge` 에는 없음.

**자세한 이슈**: `/home/vimkim/gh/my-cubrid-jira/issues/CBRD-26735-view-merge-use-merge-dropped.md`

### F8. [OPTIMIZER] 서브쿼리 `RECOMPILE` 이 view merge 로 outer 에 누설 — CBRD-26736

**배경**: 서브쿼리에만 걸린 `/*+ RECOMPILE */` 이 view merge 후 merged outer query 에 그대로 전파되어 `SHOW TRACE` 의 `Query Plan:` 및 `rewritten query:` 섹션이 통째로 사라진다. DBA / 개발자의 plan 디버깅 체인을 깨뜨린다.

**재현**:
```sql
SET TRACE ON;

-- Baseline: outer RECOMPILE → Query Plan / rewritten query 정상 출력
SELECT /*+ RECOMPILE */ * FROM (SELECT a.v FROM t1 a, t2 b WHERE a.id=b.id) v;
SHOW TRACE;

-- Bug: subquery RECOMPILE only → Query Plan / rewritten query 없음
SELECT * FROM (SELECT /*+ RECOMPILE */ a.v FROM t1 a, t2 b WHERE a.id=b.id) v;
SHOW TRACE;
-- 출력은 Trace Statistics: 섹션만 남고 plan/rewritten 사라짐
```

**Root cause**: `src/parser/view_transform.c:14724` `mq_copy_sql_hint()` 의 hint 비트 병합이 bitwise OR 로 **무조건 병합**:
```c
dest_query->info.query.q.select.hint =
    (PT_HINT_ENUM) (dest_query->info.query.q.select.hint | src_query->info.query.q.select.hint);
```
`PT_HINT_RECOMPILE` (`1ULL << 8`) 이 exclusion set 에 없어 subquery 에서 outer 로 leak. `PT_HINT_ORDERED` 는 line 14707 부근에서 올바르게 strip 되므로 동일 패턴을 `PT_HINT_RECOMPILE` 에도 적용해야 함.

**자세한 이슈**: `/home/vimkim/gh/my-cubrid-jira/issues/CBRD-26736-view-merge-recompile-leak.md`

**비고**: F7 (CBRD-26735) 와 F8 (CBRD-26736) 은 동일 함수 `mq_copy_sql_hint()` 주변의 이웃 버그이므로 동일 PR 로 묶어 수정 가능.

### F9. [PL/SP] PL/CSQL 내부 COMMIT 이 caller 의 outer 트랜잭션을 silent commit — CBRD-26742

**배경**: PL/CSQL 저장 프로시저 바디에 `COMMIT;` 문을 포함한 SP 를 active outer txn 에서 호출하면, **SP 의 COMMIT 이 outer txn 전체를 함께 커밋**한다. 이후 caller 의 `ROLLBACK;` 은 "Execute OK" 로 반환되지만 실제 undo 되는 것은 없다. **트랜잭션 원자성 침해** + **silent data corruption** 리스크.

**재현 (100%)**:
```sql
-- Setup
CREATE TABLE tA (x INT);
INSERT INTO tA VALUES (1);
COMMIT;

CREATE OR REPLACE PROCEDURE sp_just_commit()
AS
BEGIN
  COMMIT;
END;

-- Attack (csql --no-auto-commit)
UPDATE tA SET x=42;
CALL sp_just_commit();   -- SP 내부 COMMIT
ROLLBACK;                -- undo 의도

SELECT x FROM tA;        -- 42 (기대: 1)
```

**실행 순서**:
1. Outer txn `UPDATE tA SET x=42;` — 미커밋
2. `CALL sp_just_commit()` — SP 가 `COMMIT;` 실행
3. CUBRID 는 current transaction context 전체를 커밋 → outer 의 UPDATE 포함
4. `ROLLBACK;` — 미커밋 변경 없음 → silent success
5. `SELECT x;` → 42

**Impact**: 애플리케이션이 `try/catch + ROLLBACK` 패턴을 쓰더라도 CALL 된 SP 안에 `COMMIT` 이 하나라도 있으면 부분 커밋 저장 가능. 에러 / warning 없음.

**수정 방향** (단기):
- SP 내부 `COMMIT` / `ROLLBACK` 을 active outer txn 존재 시 runtime error (`ER_SP_COMMIT_IN_NESTED_TXN` 신규) 로 reject.

**수정 방향** (장기):
- Oracle 호환 `PRAGMA AUTONOMOUS_TRANSACTION` 도입하여 SP 자체 TDES 와 caller 의 TDES 격리.

**참고 소스 영역**: `src/sp/` (JNI 브리지), `pl_engine/` (Java PL 서버), `src/transaction/` (COMMIT 디스패치).

**자세한 이슈**: `/home/vimkim/gh/my-cubrid-jira/issues/CBRD-26742-sp-commit-outer-txn.md`

### A1. F1 — 근본 원인 좌표 특정

F1 (CBRD-26727) 의 실제 defect site 는 **CBRD-26652 에서 고친 `pt_coerce_expression_argument` 가 아니라 `pt_upd_domain_info()`** 이다.

- 파일: `src/parser/type_checking.c`
- 함수: `pt_upd_domain_info(parser, arg1, arg2, op, common_type, ...)` at line 11245
- 주석: _"preparing a node for a binary operation"_
- Hardcoded bug site:
  ```c
  // line 11292~11297
  else if (arg1 && arg1->type_enum == PT_TYPE_NUMERIC)
    {
      arg1_prec     = DB_DEFAULT_NUMERIC_PRECISION;   // ← 15 하드코딩
      arg1_dec_prec = DB_DEFAULT_NUMERIC_SCALE;
      arg1_units    = 0;
  ```

이 함수는 **comparison operator (PT_GT, PT_EQ, PT_LE …) / BETWEEN / CASE** 등의 argument precision 을 결정할 때 호출된다. `arg->data_type == NULL` 이면 `DB_DEFAULT_NUMERIC_PRECISION`(15) 로 폴백하므로, 컬럼의 실제 precision 이 NUMERIC(38,0) 이나 NUMERIC(20,5) 이어도 comparison 에서는 (15,9) 취급된다.

반면 CBRD-26652 에서 수정한 `pt_coerce_expression_argument()` 는 주로 **arithmetic operator (PT_PLUS / PT_MINUS / PT_TIMES / PT_DIVIDE)** 경로에서 호출되므로 산술은 정상 동작.

### A4. F2 — 추가 증거 (CTE / 중첩 inline view)

CBRD-26728 의 `mq_copy_sql_hint()` gate bug 는 **CTE 와 중첩 inline view** 에서도 동일하게 관측.

```sql
-- CTE 형태
WITH V AS ( SELECT /*+ ORDERED */ a.v FROM t1 a, t2 b WHERE a.id = b.id )
SELECT /*+ recompile */ * FROM V;
-- rewritten: select /*+ ORDERED */ V.v from [dba.t1] V, [dba.t2] b where (V.id=b.id)

-- 중첩 inline view (2-deep)
SELECT /*+ recompile */ * FROM
  ( SELECT * FROM ( SELECT /*+ ORDERED */ a.v FROM t1 a, t2 b WHERE a.id = b.id ) iv ) v;
-- rewritten: select /*+ ORDERED */ v.v from [dba.t1] v, [dba.t2] b where (v.id=b.id)
```

두 경우 모두 ORDERED 힌트가 병합된 query 로 전파된다. CBRD-26728 의 fix 가 CTE/다중 레벨 머지에도 적용되어야 한다는 점을 명시 필요.
