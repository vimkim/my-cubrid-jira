# [CBRD-26668] [OOS] Vacuum 시 OOS 레코드 정리 연동

PR: [CUBRID/cubrid#6986](https://github.com/CUBRID/cubrid/pull/6986)
Parent: [CBRD-26583](http://jira.cubrid.org/browse/CBRD-26583)
Main tracking: [CBRD-26517](http://jira.cubrid.org/browse/CBRD-26517)

---

## TL;DR

OOS 레코드는 한 번 만들어지고 나면 영영 회수되지 않아 OOS 파일이 무한 누적된다. heap 슬롯 자체는 vacuum 이 정리하지만, 그 슬롯이 가리키던 OOS 레코드는 아무도 안 지운다는 누락이다. 본 PR 은 vacuum 의 두 경로 (REMOVE / forward-walk) 와 SA_MODE 한정 eager cleanup 한 군데, 그리고 직렬화 측 두 파일에 걸친 한 가지 prerequisite (4-byte VOT alignment) 를 보강해 회수 경로를 닫는다. 새 WAL 레코드 타입은 도입하지 않고, 각 경로가 기존 또는 새로 여는 sysop 경계로 둘러싸여 `oos_delete` 호출이 atomic 하게 동작한다.

---

## Description

### 배경

OOS (Out-of-row Overflow Storage) M1 설계는 다음과 같다.

- **DELETE/UPDATE 시점**: heap 레코드에 MVCC `delete_mvccid` 만 표시한다. OOS 레코드는 즉시 삭제하지 않는다 — 동시에 다른 트랜잭션이 과거 버전을 읽고 있을 수 있기 때문.
- **vacuum 시점**: 모든 트랜잭션이 더 이상 그 버전을 못 보게 됐을 때 vacuum 이 heap 레코드를 슬롯에서 제거한다. **이때 그 레코드가 가리키던 OOS 레코드도 같이 제거되어야 한다.**

### 문제

두 번째 단계의 OOS 정리 로직이 vacuum 쪽에 구현되어 있지 않았다. 결과: OOS 레코드는 한 번 만들어지면 영원히 안 지워지고 OOS 파일이 무한 누적된다.

### 해결 방향

vacuum 의 두 경로 + SA_MODE 한정 보강 한 군데 + 직렬화 측 prerequisite 한 가지를 손봐 OOS 회수 경로를 완성한다.

1. **REMOVE 경로** (`vacuum_heap_record`): 살아 있는 heap 슬롯이 vacuum 대상이 됐을 때 그 슬롯이 가리키는 OOS OID 들을 함께 삭제.
2. **forward-walk 경로** (`vacuum_process_log_block`): UPDATE 로그를 정방향으로 만났을 때 undo 에만 남아 있는 옛 pre-image OOS OID 들을 삭제.
3. **SA_MODE eager cleanup** (`heap_update_home_delete_replaced_oos`): SA_MODE 한정 — UPDATE 시점에 옛 OOS 를 즉시 회수.
4. **VOT 4-byte alignment** (`transform_cl.c`, `catalog_class.c`): variable offset table 항목을 4-byte 정렬해 `OR_GET_VAR_OFFSET()` 의 하위 2 비트 마스킹이 OOS 플래그 비트를 보존한다.

새로운 WAL 레코드 타입은 도입하지 않는다.

## Implementation

### 1. REMOVE 경로 — `vacuum_heap_record`

`VACUUM_HEAP_HELPER` 에 `oos_vfid` 필드를 추가하고, `vacuum_oos_find_vfid_for_heap_record()` 가 `REC_HOME` / `REC_RELOCATION` 경로에서 lazy 로 채운다 (OOS 비트가 없으면 lookup 자체를 skip; `vacuum.c:2074, 2189`).

레코드 타입별 처리:

- **REC_HOME + OOS** (`vacuum.c:2616-2619`): `vacuum_heap_record_remove_oos_inline()` 헬퍼가
  1. heap 슬롯 vacuum 로그 기록 (`vacuum_log_redoundo_vacuum_record`)
  2. 슬롯의 OOS OID 들을 `oos_delete` 로 일괄 삭제 (`vacuum_heap_oos_delete`)
  3. `log_sysop_commit` (실패 시 `log_sysop_abort`)
  를 **하나의 sysop 으로 묶는다**.

  > bulk vacuum 경로 대신 inline-log 를 쓰는 이유: bulk 경로는 heap-slot vacuum 로그가 sysop 밖에서 emit 되므로, sysop commit 후 bulk flush 전 crash 시 OOS 는 사라졌는데 heap 슬롯은 여전히 그 OOS 를 가리키는 상태가 된다. inline-log 는 양쪽을 같은 sysop 경계에 묶어 이를 방지.

- **REC_RELOCATION + OOS** (`vacuum.c:2562-2573`): 기존 sysop 내에서 `log_sysop_commit` 직전에 `vacuum_heap_oos_delete` 호출.
- **REC_BIGONE** (`vacuum.c:2587`): `assert (!heap_recdes_contains_oos)` — design decision: OOS 의 목적이 레코드를 작게 유지하는 것이므로 INSERT/UPDATE 가 OOS-bearing recdes 를 REC_BIGONE 으로 보내지 않는다. 다만 현재는 debug-only assert 로만 enforce 되며 — 현 위치의 `assert (...)` 를 `assert_release (...)` 로 단순 치환해 release-build 에서도 enforce 하는 것은 scope-out (Remarks 참조; 추가 검증 로직이 아니라 같은 위치의 매크로 교체).

### 2. forward-walk 경로 — `vacuum_process_log_block`

vacuum 은 어차피 모든 MVCC 로그를 정방향으로 훑는다. `vacuum.c:3690` 의 게이트가 `RVHF_UPDATE_NOTIFY_VACUUM` 만 골라내, 그 undo 페이로드(= UPDATE 직전의 recdes)를 디코드하고 OOS 비트가 켜져 있으면 이전 버전이 가리키던 OOS OID 들을 `vacuum_forward_walk_delete_old_oos()` (`vacuum.c:3455`) 헬퍼로 일괄 삭제한다. forward-walk 진행 시점은 vacuum 의 `threshold_mvccid` 가드 너머이므로 옛 OID 가 가리키던 OOS body 는 더 이상 어떤 reader 에게도 보이지 않는다 — MVCC visibility 측면에서 회수가 안전.

> **forward-walk 단위 실패는 vacuum block 을 중단시키지 않는다** (`vacuum.c:3711-3716`): `heap_recdes_get_oos_oids` 또는 `vacuum_forward_walk_delete_old_oos` 가 non-zero 를 반환하면 warning 로깅 + `er_clear()` 후 다음 로그 레코드로 진행한다. 의도: 한 chunk 의 회수 실패가 같은 block 안 다른 OID 의 회수까지 막지 않게 하는 것. 단점: 실패한 OID 는 다음 vacuum 사이클에서 재시도되지 않으므로 leak 가능 (design accepted; 회수 보장이 아니라 best-effort cleanup).

#### 핵심 원리: forward-walk 는 "REMOVE 경로가 못 닿는 옛 OID" 의 *유일한* 청구권자 (SERVER_MODE 한정)

REMOVE 경로(§1)는 **현재 살아 있는 슬롯이 가리키는** OOS OID 만 본다. forward-walk 경로(§2)는 **undo 에만 남아 있는 옛 pre-image OID** 를 본다. 두 경로가 같은 OID 를 두 번 보지 않는 것은 다음 invariant 덕분이다.

> **UPDATE OID-disjointness invariant**: `heap_attrinfo_insert_to_oos` (`heap_file.c:12368`) 는 매 transform 마다 `oos_insert` 로 새 OOS OID 를 신규 할당한다 (`heap_file.c:12421`). 따라서 UPDATE 직후 heap 슬롯의 recdes 는 **새 OID** 만 들고 있고, pre-image 의 옛 OID 는 그 어떤 살아 있는 heap 위치에서도 참조되지 않는다.

따라서 SERVER_MODE 에서 옛 OID 는 REMOVE 경로가 영원히 못 닿고, forward-walk 가 유일한 회수 경로다. SA_MODE 에서는 §3 의 eager cleanup 이 동일 OID 를 UPDATE 시점에 미리 회수하므로 forward-walk 는 사실상 no-op 이다.

#### 다른 MVCC 로그 종류는 모두 제외 — 두 가지 차단 메커니즘

| rcvindex | undo payload | 같은 OID 를 다른 경로가 회수? | rcvindex 발행 지점 (실제 `log_append_*`) | 결정 |
|---|---|---|---|---|
| `RVHF_UPDATE_NOTIFY_VACUUM` | undo recdes 보유 (UPDATE pre-image) | **X** (post-image 가 fresh OID) | `heap_file.c:24537, 24541` (`heap_log_update_physical` 내 `log_append_undo_recdes` / `log_append_undoredo_recdes`; 호출자 `:23742, :23954` 가 `is_mvcc_op ? RVHF_UPDATE_NOTIFY_VACUUM : RVHF_UPDATE` 결정) | **포함** |
| `RVHF_MVCC_DELETE_MODIFY_HOME` | undo recdes 보유 (`old_recdes`, REL→HOME 등 slot transition 시 pre-image 동봉) | O (REMOVE 경로가 같은/이주된 OID 를 회수) | `heap_file.c:20750/20754` (`heap_mvcc_log_home_change_on_delete`, `no_logging` vs 일반 분기) | **명시 제외** |
| `RVHF_MVCC_INSERT` | undo size 0, no crumbs | n/a (INSERT 의 undo 는 슬롯 제거뿐) | `heap_file.c:17468, 17472` | undo_data 가드로 자동 차단 |
| `RVHF_MVCC_DELETE_REC_HOME` | undo size 0 (`heap_mvcc_log_delete`, no crumbs in both branches) | n/a | `heap_file.c:17691` (`no_logging`, `log_append_undo_data`), `heap_file.c:17695` (`log_append_undoredo_data`) | undo_data 가드로 자동 차단 |
| `RVHF_MVCC_NO_MODIFY_HOME` | undo size 0, no crumbs | n/a | `heap_file.c:20779` | undo_data 가드로 자동 차단 |
| `RVHF_MVCC_REDISTRIBUTE` | undo size 0, no crumbs (`undoredo_crumbs (..., 0, n_redo, NULL, ...)`) | n/a | `heap_file.c:26418` | undo_data 가드로 자동 차단 |

> **REC_BIGONE 과 OOS 의 관계 (참고)**: §1 의 invariant 는 *"BIGONE 의 현재 recdes 에는 OOS 가 없다"* 를 가리키는 design decision 이다. 반면 위 표의 `RVHF_MVCC_DELETE_MODIFY_HOME` 처럼 *transition 로그의 undo recdes 에는* OOS 비트가 있을 수 있다 (REC_HOME → REC_BIGONE 으로 가는 시점의 pre-image).

**`rcvindex == RVHF_UPDATE_NOTIFY_VACUUM` 가드가 *load-bearing* 한 케이스는 단 하나, `RVHF_MVCC_DELETE_MODIFY_HOME` 차단이다.** 나머지 4 종은 `undo_data != NULL && undo_data_size > 0` 만으로도 자연 차단된다. 향후 새 MVCC heap rcvindex 가 도입되면 (특히 undo recdes 를 들고 가는 종류) 이 가드의 차단 범위를 다시 검토해야 한다.

#### `RVHF_MVCC_DELETE_MODIFY_HOME` 가 까다로운 이유

`heap_mvcc_log_home_change_on_delete` (`heap_file.c:20732`) 는 docstring 그대로 **"REC_HOME → REC_HOME 으로 끝나지 않는 MVCC delete"** 시 호출된다 — 즉 REC_RELOCATION 이 home 으로 돌아오거나, REC_HOME 이 REC_RELOCATION/REC_BIGONE 으로 변환되는 슬롯-타입 전이 상황. 이 로그가 들고 가는 `old_recdes` 는 OOS bit + OOS OID 를 그대로 보유한다. 그리고 transition 결과로 OOS body 는 사라지지 않고 새 위치(이주한 home / forwarding 슬롯)에 그대로 살아 있어 **REMOVE 경로가 결국 그 OID 들을 회수** 한다. 따라서 forward-walk 가 `old_recdes` 를 같이 손대면:

```
REMOVE 경로:    1차 oos_delete(OID_X) OK → chunk chain 해제
forward-walk:  2차 oos_delete(OID_X) → S_DOESNT_EXIST assert → vacuum block 실패
```

UPDATE 와 대비: UPDATE 는 post-image 가 fresh OID 로 다시 작성되므로, 슬롯이 같은 위치에 머물더라도 옛 OID 는 그 어디에도 살아 있지 않고 REMOVE 경로가 옛 OID 에 닿을 길이 없다.

#### Log-page rotation 안전성 (OID 벡터 사전 복사로 충분)

`oos_delete` 는 `RVOOS_DELETE` 를 chunk 마다 append 하면서 log page 를 회전시킬 수 있어, 원래 `undo_data` 포인터가 가리키는 페이지가 무효화될 수 있다. 본 PR 은:

- 헬퍼 진입 **전에** `heap_recdes_get_oos_oids` 가 OID 들을 self-owned `OID_VECTOR` 로 한 번에 복사
- 이후 `oos_delete` 루프는 그 벡터만 참조하지 `undo_data` 를 다시 안 읽음

따라서 헬퍼 내부에 추가적인 방어 복사 없이 안전하다.

#### Block-local OOS VFID 캐시

vacuum block 안에서 같은 heap 파일의 여러 레코드가 처리될 수 있다. heap VFID → OOS VFID 매핑을 매번 file tracker 로 조회하면 비싸므로 block 단위로 16-슬롯 stack-local 배열 캐시를 둔다 (`vacuum.c:3526`). lookup 실패(`HFID_IS_NULL`, `file_descriptor_get` 실패, `heap_oos_find_vfid` 실패)는 캐시에 기록하지 않아 다음 호출에서 재시도된다 (false negative cache 회피).

> **False-positive 리스크 가정**: heap VFID → OOS VFID 매핑은 heap 파일의 lifetime 동안 불변이라고 가정한다 (OOS 파일은 heap 생성 시 `heap_create_internal` 에서 함께 만들어지고 heap drop 까지 유지). 같은 vacuum block 안에서 lookup 성공 후 그 매핑이 무효화되는 경로는 없다 (heap drop 은 vacuum 중에 진행되지 않음). 따라서 캐시 히트는 항상 유효.

### 3. SA_MODE eager cleanup — `heap_update_home_delete_replaced_oos`

SA_MODE 에서도 vacuum 은 동작하지만 — 단일 프로세스, 동시 reader 없음 — **forward-walk 가 해당 로그까지 진행하기 전에 동일 트랜잭션의 후속 INSERT 가 동일 OOS 파일에서 page 재할당을 시도할 수 있다**. INSERT 는 `oos_find_best_page` 의 best-space lookup 으로 free chunk 를 찾는데, 옛 chunk 가 아직 살아 있으면 free-space accounting / chunk chain 이 inconsistent 한 상태에서 새 chunk 가 끼어들 수 있다. 따라서 UPDATE 시점에 옛 OOS 를 즉시 회수해야 한다 (optimization 이 아니라 **correctness 보강**).

`heap_update_home_delete_replaced_oos()` (`heap_file.c:24131`) 가 pre-image (= 갱신 전 home page recdes, `context->home_recdes`) / post-image (= 새로 작성한 recdes, `context->recdes_p`; 두 필드 모두 `HEAP_OPERATION_CONTEXT` 의 정의는 `heap_file.h:266-285` 참조) 의 OID 차집합을 계산해 `oos_delete` 를 즉시 호출. 양쪽에 있는 OID (= 같은 OOS 를 그대로 재참조) 는 `heap_oos_oid_in_vector` 검사로 보존 (`heap_file.c:24180`). 호출 위치는 `heap_update_home` 의 non-MVCC 분기 (`heap_file.c:24416-24424`).

### 4. VOT 4-byte alignment — `transform_cl.c`, `catalog_class.c`

class record / catalog 직렬화 시 variable-data 시작 위치를 4-byte 로 정렬한다. `OR_GET_VAR_OFFSET()` 가 하위 2 비트를 마스킹해 OOS 플래그 등을 보존하므로, 정렬되지 않은 offset 은 마스킹 시 OOS 플래그 비트가 손실된다.

- `catalog_class.c`: `catcls_align_var_data_to_4()` 헬퍼 (`catalog_class.c:3201`) 가 per-attribute (`:3313`) / 마지막 offset (`:3326`) 두 경로에서 공통으로 사용된다.
- `transform_cl.c`: 별도 in-place 패딩 — 고정 폭 블록 종료 후 (`:736`) / variable-data 시작 정렬 (`:755-758`) / 가변 attribute 사이 (`:766-769`) 에서 `DB_ALIGN (written, 4)` + `or_pad` 로 다음 variable-data 시작점을 4-byte 로 맞춘다. `catcls_*` 헬퍼는 catalog_class 전용이라 transform_cl 측은 동일 alignment 를 인라인으로 구현한다.

### 5. Crash recovery (새 WAL 타입 없음)

- **per-chunk `RVOOS_DELETE` undoredo**: `oos_delete` 가 chunk 마다 이미 기록 — redo 정상 replay (`oos_file.cpp:1698`, `log_append_undoredo_recdes (..., RVOOS_DELETE, ...)`).
- **sysop 원자성**: REMOVE / forward-walk 모두 `log_sysop_start ~ log_sysop_commit/abort` 로 묶임 — sysop commit 이전 crash 시 전체 sysop 자동 롤백 (heap 슬롯 / OOS 청크가 같이 변경 전 상태로 복귀).
- **multi-chunk `oos_delete` 도중 process death**: 표준 ARIES 보장 — recovery 가 in-progress sysop 을 unwind 하면서 이미 emit 된 per-chunk `RVOOS_DELETE` 들을 역순으로 undo 하여 partial chunk 변경을 보상. 즉 forward-walk / REMOVE 헬퍼 안에서 한 sysop 이 N 개의 chunk 를 반쯤 지운 채 죽어도 sysop 단위 unwind 가 일관성을 복구한다.
- **sysop 1:1 매칭 검증**: outer `is_under_sysop` 가드 5 군데 (`vacuum.c:3537` 진입, `:3858` / `:3862` 중간, `:3871` / `:3879` 종료) 가 `vacuum_process_log_block` 진입/구간/종료에서 누수를 차단. sysop start/commit/abort triple 은:
  - **forward-walk 헬퍼**: `vacuum_forward_walk_delete_old_oos` 안에 1 세트 (start at `vacuum.c:3460`, commit at `:3471`, abort at `:3475`) 로 닫힘.
  - **REMOVE inline**: 헬퍼 `vacuum_heap_record_remove_oos_inline` 자체는 commit (`vacuum.c:2463`) / abort (`:2459`) 만 들고 있고, **outer `log_sysop_start` 는 caller `vacuum_heap_record` 에 있다** (`vacuum.c:2492`). 즉 forward-walk 와 달리 inline 은 자체 sysop 을 열지 않고 caller 의 sysop 경계를 공유한다.
- redo handler 에서 추가 로그 append 안 함 → double-replay hazard 없음.

## Acceptance Criteria

- [x] vacuum 의 REMOVE 경로가 OOS 정리 (REC_HOME / REC_RELOCATION + OOS)
- [x] forward-walk 경로가 UPDATE pre-image OOS 정리 (`RVHF_UPDATE_NOTIFY_VACUUM` 한정)
- [x] forward-walk 가 `RVHF_MVCC_DELETE_MODIFY_HOME` 을 명시적으로 제외 (이중 삭제 방지)
- [x] SA_MODE eager cleanup
- [x] VOT 4-byte alignment
- [x] block-local 16-slot OOS VFID 캐시

## Remarks

### Scope-out (별도 JIRA 추적)

- **REC_BIGONE + OOS invariant 강화**: 현재 debug-only `assert (!heap_recdes_contains_oos)`. release-build 에서도 강제하는 `assert_release` 전환은 후속.
- **Crash injection 회귀 테스트**: design-level sysop 보장은 입증되지만 자동 crash injection 테스트는 본 이슈 범위 밖.
- **VFID 캐시 자료구조**: 현재 16-slot linear scan + slot 0 overwrite. partition-heavy 워크로드에서 thrashing 가능 → perf-driven 후속에서 `unordered_map` / open addressing 검토.
- **`RVOOS_NOTIFY_VACUUM` rcvindex**: 등록은 되어 있으나 emitter 없음. eager-vacuum 신호용 reserve 상태 — 추가/제거 결정은 후속.
- **forward-walk OID 리스트 정렬**: heap 의 buffer-pool locality 패턴 따라하기 — 후속 perf 최적화.
- **forward-walk + DELETE_MODIFY_HOME 더블-삭제 시나리오 직접 재현 회귀 테스트**: 후속.
- **`OosVacuumCodePathServer.DISABLED_PrevVersionChainWalker*`**: 이전 backward-chain 설계의 잔재 테스트 — forward-walk 전환 후 의도적으로 disabled 유지. 재활성화는 후속.

### 성능 회귀 분석 (해결 완료)

이전 중간 단계의 `prev_version_lsa` backward chain walker 설계는 `sql/_01_object/_09_partition/_001_create/cases/1189.sql` 에서 회귀를 일으켰다 (커밋 `0748432eb`). 현재 forward-walk inline 설계로 재구성하면서 회귀 자체가 해소됐다 — vacuum 이 어차피 forward-walk 하는 비용에 편승하므로 추가 cost path 가 없다.
